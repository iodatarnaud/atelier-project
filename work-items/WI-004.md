## CADRAGE

Goal:
Empêcher la perte silencieuse de données quand l'app Atelier est ouverte sur 2 machines (ou 2 onglets) en parallèle. Aujourd'hui, une instance qui modifie après qu'une autre a pushé écrase aveuglément le Gist (last-write-wins violent, sans détection ni signal).

Persona impact:
- Arnaud qui jongle entre Mac et Windows pendant la même journée.
- Arnaud qui laisse un onglet ouvert sur une machine et travaille sur une autre.
- Tout futur cas multi-onglet (même machine, fenêtres séparées).

Context:
- Item backlog : ATE-22 « Sync multi-instances » (Bug, P1, projet Atelier).
- **Symptôme observé (Arnaud, 2026-04-30)** : Mac ouvert sur Atelier, modifs faites sur Windows pendant ce temps, retour sur Mac → modifs Windows invisibles.
- **Audit code (Claude, hors WI)** :
  - Pull du Gist : **uniquement au boot** dans `loadState()` ([`index.html:2823-2851`](index.html#L2823-L2851)). Comparaison `if (remoteTs > localTs)` correcte mais one-shot.
  - Push : `scheduleGistPush()` ([`index.html:2735-2747`](index.html#L2735-L2747)) avec debounce 2,5s, **pas de comparaison avec remoteTs distant avant écrasement**.
  - `beforeunload` : tente un push synchrone si timer en attente ([`index.html:2750-2772`](index.html#L2750-L2772)). N'aide pas pour le cas concurrent.
  - **Aucun re-fetch périodique, aucun listener `focus` / `visibilitychange`, aucun garde push-time.**
  - Politique « last-write-wins via `lastSavedAt` » documentée dans `CLAUDE.md` est cohérente entre boots successifs, **pas entre sessions concurrentes ouvertes**.
- Scénarios à risque :
  - **R1** : Mac ouvert + Windows pushe + Mac modifie sans refresh → push Mac écrase Windows. **Perte de données.**
  - **R2** : Mac ouvert + Windows pushe + Mac n'a rien modifié → désync visuelle (Mac affiche un état périmé jusqu'au refresh). Pas de perte mais surprise UX.

Constraints:
- Pas de migration de schéma `lastSavedAt` (champ existant suffit).
- Préserver la compatibilité Gist : le format JSON pushé reste identique (toujours `{ clients, activeClientId, lastSavedAt }`).
- Pas de backend de coordination (toujours zéro serveur tiers).
- Conserver le mode test (`atelier-mode-test`) : aucune écriture distante, donc aucun garde sync ne s'active.
- Rate limit GitHub API : 5000 requêtes/heure par PAT — large marge mais éviter le polling agressif inutile.
- Pas d'introduction de tokens / locks / ETags GitHub avancés (Gist API ne supporte pas vraiment) — on s'appuie uniquement sur `lastSavedAt`.

Out of scope:
- Vraie résolution de conflit field-by-field (CRDT, OT). Trop coûteux pour un mono-utilisateur.
- Verrouillage distant (locking) — Gist API ne le supporte pas natif.
- Pull périodique automatique systématique (consomme rate limit pour peu de gain si focus + push-guard couvrent les cas critiques — à arbitrer en CADRAGE).
- Refonte du flow `loadState` / `saveState` au-delà des points d'entrée sync.
- Audit Trail / historique des merges (l'activité par item existe déjà via timeline events, mais pas de log inter-session).

Open questions:

1. **Stratégie de pull additionnel après le boot** — comment détecter les modifs distantes pendant que l'app est ouverte ?
   - **(A) Pull au focus / visibilitychange** : quand l'onglet/fenêtre redevient visible, refetch Gist et applique si plus récent. Pattern Slack/Notion. Économe en rate limit. Couvre 90 % du cas réel.
   - **(B) Pull périodique** : toutes les N secondes (30 / 60 / 120). Plus systématique, consomme du rate limit. Couvre aussi le cas "tu reviens sur la fenêtre déjà visible".
   - **(C) Combinaison A + B** : focus pour réactif, périodique en filet de sécurité.
   - **(D) Pull manuel uniquement** : bouton "↻ Sync now" dans le topbar, pas d'auto. Donne le contrôle mais repose sur la mémoire du user.

2. **Push-guard (pull avant push)** — avant chaque `pushGist`, doit-on refetch et comparer ?
   - **(a) Oui systématique** : refetch, si `remoteTs > snapshotLocalTs` (snapshot pris au moment de la modif locale) → conflit détecté, ne pas écraser, basculer sur Q3.
   - **(b) Non, on garde le push aveugle** : statu quo dégradé, on accepte le risque pour la simplicité.
   - **(c) Configurable utilisateur** : option dans Paramètres « Vérifier avant chaque push (recommandé) ». Défaut = oui.

3. **Comportement si conflit détecté** (modif distante + modif locale non pushée) :
   - **(i) Toast avec choix manuel** : "⚠ Données distantes plus récentes — [Recharger] [Garder local] [Voir diff]". Le user décide.
   - **(ii) Bloquer le push, demander à l'utilisateur** sans toast (modale bloquante). Plus sûr mais intrusif.
   - **(iii) Auto-merge naïf** : appliquer remote, perdre local (équivaut à statu quo dans l'autre sens).
   - **(iv) Last-write-wins assumé** même en conflit, juste afficher un toast informatif "✓ Sync OK (la version la plus récente a gagné)".

4. **Indicateur visuel quand un pull silencieux ramène du nouveau** (sans conflit, juste mise à jour normale) :
   - **(α) Toast court** : "↺ Données mises à jour (depuis [autre machine])"
   - **(β) Re-render silencieux** : pas de notif, le user voit dans l'UI s'il regarde
   - **(γ) Badge sur l'indicateur de sync** : pulse rapide ou couleur changeante 2s

Decisions:
1. **Q1 → (A) Pull au focus / visibilitychange.** Justif : couvre 95 % du flow réel (Mac fermé/ouvert/visible vs background) sans consommer de rate limit GitHub inutile. Pattern Slack/Notion éprouvé.
2. **Q2 → (a) Push-guard systématique.** Justif : Arnaud travaille parfois en simultané sur 2 ordis (Mac + Windows visibles en même temps), donc le focus seul ne suffit pas. Bretelles indispensables pour 100 % de couverture, coût négligeable (1 GET en plus avant chaque PUT).
3. **Q3 → (i) Toast avec choix manuel** [Recharger | Garder local | Voir diff]. Justif : pattern non-bloquant qui respecte le flow de saisie tout en signalant le conflit. Le user décide en pleine conscience, pas de perte invisible.
4. **Q4 → (α) Toast court** "↺ Données mises à jour (depuis une autre machine)". Justif : cohérent avec Q3 (toast aussi), évite tout changement d'état "mystérieux" sous les yeux du user. Disparaît en 3-4s, non-bloquant.

Bloc retenu : **A + a + i + α** = approche "pessimiste UX" : on protège les données, on signale tout, on ne bloque rien.

Note canal : décisions transmises par Arnaud via chat après échange story-driven (1 question, 1 narrative concrète), inscrites par Claude dans la section. Pattern récurrent depuis WI-003 — à observer en RETROSPECTIVE pour formaliser ou tolérer.

## PRD

Révision après retour PLAN_REVIEW (re-cadrage)

Objective:
Implémenter le set de garde-fous **A + a + i + α** validé en `CADRAGE.Decisions` pour empêcher la perte silencieuse de données entre 2 sessions Atelier ouvertes en parallèle (Mac + Windows ou multi-onglets).

Current behavior:
- Pull du Gist : **uniquement au boot** dans `loadState()` ([`index.html:2823-2851`](index.html#L2823-L2851)).
- `scheduleGistPush()` ([`index.html:2735-2747`](index.html#L2735-L2747)) PATCH le Gist (debounce 2,5s) sans vérifier que la version distante n'a pas changé entre temps → écrasement aveugle.
- `beforeunload` ([`index.html:2750-2772`](index.html#L2750-L2772)) émet un PATCH `keepalive` direct si `_gistPushTimer` est en attente — sans push-guard possible (GET async non fiable en unload).
- `connectGist()` ([`index.html:2935-3004`](index.html#L2935-L3004)) gère un confirm dialog local-vs-remote au moment de l'opt-in initial — flow indépendant de la sync running.
- `showToast(msg)` ([`index.html:3043-3055`](index.html#L3043-L3055)) : toast simple non-bloquant, 3s d'auto-dismiss, pas de boutons d'action.
- Aucun listener `focus` / `visibilitychange`. Aucun toast persistant à actions. Aucun tracking de modif locale non pushée.

Target behavior:

**État mémoire sync (nouveau)** — 4 variables module-level :
- `_lastSyncedRemoteTs` (number, init 0) : `lastSavedAt` du dernier payload remote synchronisé (boot, pull focus appliqué, push réussi, push forcé). C'est la baseline de comparaison du push-guard.
- `_localDirtySinceTs` (number ou null, init null) : timestamp de la première modif locale invalidant `_lastSyncedRemoteTs`. Set à `Date.now()` dans `saveState()` (mode non-test, sync configurée) si null. Reset à null après push réussi ou apply remote.
- `_syncConflict` (objet ou null, init null) : `{ remote, remoteTs, detectedAt }` quand un conflit est ouvert. Null sinon. Bloque les pushs supplémentaires tant qu'il est non-null.
- `_syncCheckInFlight` (boolean, init false) : verrou pour dédupliquer un GET (focus ↔ visibilitychange ↔ push-guard quasi-simultanés).

**Pull au focus / visibilitychange (A)** — nouveau listener attaché à `window.addEventListener('focus', …)` ET `document.addEventListener('visibilitychange', …)`. Au déclenchement (mode non-test, sync configurée, `_syncCheckInFlight === false`) :
1. Set `_syncCheckInFlight = true`, `await fetchGist()`.
2. Si `remoteTs <= _lastSyncedRemoteTs` → no-op silencieux, release flag.
3. Si `remoteTs > _lastSyncedRemoteTs` ET `_localDirtySinceTs === null` (rien de dirty) → `applyRemoteState(remote, remoteTs, { toast: 'info' })` + toast info **(α)**.
4. Si `remoteTs > _lastSyncedRemoteTs` ET `_localDirtySinceTs !== null` (local dirty) → `_syncConflict = { remote, remoteTs, detectedAt: Date.now() }` + toast conflit **(i)**, MÊMES actions qu'AC3.

**Push-guard (a)** — refactor de `scheduleGistPush()` : avant chaque PATCH, GET via `fetchGist()`.
1. Si `_syncConflict !== null` → push annulé (conflit déjà ouvert), no-op.
2. Set `_syncCheckInFlight = true`, `await fetchGist()`.
3. Erreur GET → push **bloqué**, toast erreur "⚠ Sync indisponible — réessaie plus tard.", flag relâché. `_gistPushTimer` non re-armé automatiquement (la prochaine modif user re-armera via `scheduleGistPush()`).
4. Si `remoteTs <= _lastSyncedRemoteTs` → push normal (PATCH avec `lastSavedAt: Date.now()`). Après succès : `_lastSyncedRemoteTs = <lastSavedAt du payload pushé>`, `_localDirtySinceTs = null`.
5. Si `remoteTs > _lastSyncedRemoteTs` → push **bloqué**, `_syncConflict = { remote, remoteTs, detectedAt: Date.now() }`, toast conflit **(i)**.

**Toast conflit (i)** — toast persistant avec 3 boutons. Container DOM dédié (`#syncConflictToast`) distinct de `#toast` existant. Boutons taggués `data-testid` pour Playwright. Pas d'auto-dismiss tant que `_syncConflict !== null`.
- **Recharger** → `applyRemoteState(_syncConflict.remote, _syncConflict.remoteTs, { toast: 'reloaded' })` ; `_syncConflict = null`.
- **Garder local** → `await pushGist({ ..., lastSavedAt: <ts capturé> })` direct (sans push-guard puisque user décide). Après succès : `_lastSyncedRemoteTs = <ts capturé>`, `_localDirtySinceTs = null`, `_syncConflict = null`, toast confirmation. En cas d'erreur PATCH, conflit reste ouvert.
- **Voir diff** → ouvre modale `#syncDiffModal` (AC6). `_syncConflict` reste ouvert tant que la modale n'a pas pris d'action.

**Toast info pull silencieux (α)** — toast volatil 3-4s "↺ Données mises à jour (depuis une autre machine)". Réutilise `showToast()` existant.

**Helper unique** — `applyRemoteState(remote, remoteTs, { toast })` :
1. `state.clients = remote.clients`, `state.activeClientId = remote.activeClientId || state.clients[0]?.id`.
2. `dbSet(STORAGE_KEY, { clients, activeClientId, lastSavedAt: remoteTs })`.
3. `_lastSyncedRemoteTs = remoteTs`, `_localDirtySinceTs = null`.
4. `render()`.
5. Toast selon param : `'info'` → "↺ Données mises à jour…" ; `'reloaded'` → "↺ Synchronisé avec la version distante" ; `null` → silencieux (boot).

**`beforeunload` — décision** : on **conserve** le PATCH `keepalive` existant MAIS on l'inhibe si `_syncConflict !== null`. Justif : retirer complètement le push opportuniste créerait un risque de perte des modifs en cours de debounce (2,5s) à chaque fermeture d'onglet ; le garder mais l'inhiber quand un conflit est déjà détecté évite l'écrasement aveugle dans le seul cas réellement dangereux (le user a déjà été notifié, ses modifs sont préservées en IndexedDB de toute façon). Le cas "session périmée mais aucun conflit détecté" reste un risque résiduel **assumé** — il faudrait un GET sync (impossible en unload) pour le couvrir.

**`connectGist()` (opt-in initial)** — flow existant **conservé tel quel** (confirm dialog local-vs-remote). Justif : c'est un événement one-shot, pas un cas de conflit récurrent ; le confirm dialog y est déjà adapté. À l'issue de `connectGist()` (succès) : `_lastSyncedRemoteTs = <lastSavedAt du state actif après confirm>`, `_localDirtySinceTs = null`.

User stories:
- **US1** — En tant qu'Arnaud qui jongle entre Mac et Windows, je veux que mon Mac récupère automatiquement les modifs Windows quand je reviens sur l'onglet, sans avoir à refresh.
- **US2** — En tant qu'Arnaud qui modifie depuis Mac alors que Windows a déjà pushé entre temps, je veux être alerté du conflit avec 3 options claires, plutôt que d'écraser silencieusement Windows. Le cas s'applique aussi bien au push qu'au retour de focus avec local dirty.
- **US3** — En tant qu'Arnaud face à un conflit, je veux pouvoir voir ce qui diffère (compteurs + items à titre divergent, top 10) pour décider en connaissance.
- **US4** — En tant qu'Arnaud, je veux qu'aucun toast ne s'affiche en mode test (cohérent avec le no-op global du mode test).

Acceptance criteria:

### AC1 — Pull au focus / visibilitychange (sans local dirty)
Given l'app est ouverte avec sync Gist active (mode non-test), `_localDirtySinceTs === null`, l'onglet/fenêtre perd le focus puis le récupère
When l'event `focus` ou `visibilitychange` (visible) est déclenché
Then un seul `fetchGist()` est appelé (déduplication via `_syncCheckInFlight`).
And si `remoteTs > _lastSyncedRemoteTs` → `applyRemoteState(remote, remoteTs, { toast: 'info' })` + re-render + toast info (AC7).
And si `remoteTs <= _lastSyncedRemoteTs` → no-op silencieux (pas de toast, pas de re-render).
And aucun listener n'est attaché en mode test.

### AC2 — Pull au focus avec local dirty (conflit)
Given l'app est ouverte avec sync Gist active (mode non-test), `_localDirtySinceTs !== null` (modifs locales non pushées)
When un focus / visibilitychange déclenche le GET et `remoteTs > _lastSyncedRemoteTs`
Then `_syncConflict` est armé, le push éventuellement en attente (`_gistPushTimer`) reste désarmé tant que `_syncConflict !== null`.
And toast conflit (AC4) s'affiche avec les 3 actions habituelles.
And aucun apply remote silencieux n'est exécuté.

### AC3 — Push-guard avant chaque push
Given une modif locale a déclenché `scheduleGistPush()` (debounce 2,5s écoulé), `_syncConflict === null`
When le push s'apprête à PATCH le Gist
Then un GET préalable est exécuté.
And si erreur GET → push **bloqué**, toast erreur "⚠ Sync indisponible — réessaie plus tard.", `_gistPushTimer` non re-armé, state local préservé.
And si `remoteTs <= _lastSyncedRemoteTs` → PATCH (lastSavedAt: Date.now() capturé `tsPushed`), puis `_lastSyncedRemoteTs = tsPushed`, `_localDirtySinceTs = null`.
And si `remoteTs > _lastSyncedRemoteTs` → push **bloqué**, `_syncConflict` armé, toast conflit (AC4).

### AC4 — Toast conflit avec 3 actions
Given `_syncConflict !== null` (déclenché par AC2 ou AC3)
When le toast s'affiche
Then container `#syncConflictToast` rendu avec texte "⚠ Données distantes plus récentes — depuis [date relative `_syncConflict.remoteTs`]" et 3 boutons `data-testid="sync-reload"`, `data-testid="sync-keep-local"`, `data-testid="sync-view-diff"`.
And le toast reste visible tant que `_syncConflict !== null` (pas d'auto-dismiss).
And l'app reste utilisable pendant que le toast est visible (non-bloquant).
And aucun second push n'est tenté tant que `_syncConflict !== null`.

### AC5 — Action "Recharger" du toast conflit
Given un toast conflit est affiché
When l'utilisateur clique `data-testid="sync-reload"`
Then `applyRemoteState(_syncConflict.remote, _syncConflict.remoteTs, { toast: 'reloaded' })` est invoqué.
And `_syncConflict = null`, `_lastSyncedRemoteTs = remoteTs`, `_localDirtySinceTs = null`.
And toast info "↺ Synchronisé avec la version distante" 3-4s.

### AC6 — Action "Garder local" du toast conflit
Given un toast conflit est affiché
When l'utilisateur clique `data-testid="sync-keep-local"`
Then `pushGist({ clients, activeClientId, lastSavedAt: tsPushed })` est appelé avec `tsPushed = Date.now()` capturé en variable locale.
And après succès PATCH : `_lastSyncedRemoteTs = tsPushed` (réutilise EXACTEMENT le même timestamp que le payload), `_localDirtySinceTs = null`, `_syncConflict = null`.
And toast confirmation "✓ Version locale poussée".
And si erreur PATCH → conflit reste ouvert, toast erreur, `_syncConflict` non remis à null.

### AC7 — Action "Voir diff" du toast conflit
Given un toast conflit est affiché
When l'utilisateur clique `data-testid="sync-view-diff"`
Then la modale `#syncDiffModal` s'ouvre avec :
- nb clients, nb items (somme sur tous clients), nb sprints (somme), nb epics (somme) : local vs remote, deltas en + / - (rouge / vert).
- liste des items dont l'`id` existe des deux côtés mais avec un `title` différent (max 10 affichés, ordre alphabétique de title local).
And la modale a en footer 2 boutons réutilisant les actions AC5 / AC6 : `Recharger` et `Garder local`.
And `Échap` ferme la modale sans action ; le toast conflit reste affiché derrière (`_syncConflict` inchangé).

### AC8 — Toast info pull silencieux
Given un pull au focus (AC1) ramène `remoteTs > _lastSyncedRemoteTs` ET `_localDirtySinceTs === null`
When `applyRemoteState` exécute avec `{ toast: 'info' }`
Then un toast info "↺ Données mises à jour (depuis une autre machine)" s'affiche 3-4s, non-bloquant, via `showToast()` existant.

### AC9 — `beforeunload` inhibé en conflit
Given `_syncConflict !== null` au moment du `beforeunload`
When l'event est déclenché
Then aucun PATCH `keepalive` n'est émis (le push opportuniste est inhibé).
And si `_syncConflict === null`, le PATCH `keepalive` existant est conservé tel quel (comportement historique).

### AC10 — Aucune régression sur le mode test
Given l'app est en mode test (`atelier-mode-test`)
When un événement focus/visibilitychange/save/push/beforeunload se produit
Then aucun fetch / push Gist n'est tenté, aucun toast sync n'apparaît, aucun listener focus/visibilitychange n'est attaché (cohérent avec `isTestMode()` no-op global).

### AC11 — Aucune régression sur le flow boot
Given un boot normal de l'app (chargement initial), sync Gist configurée
When `loadState()` s'exécute
Then comportement actuel préservé : IndexedDB d'abord, puis fetch Gist, puis comparaison `remoteTs > localTs` → apply remote.
And à la fin du boot : `_lastSyncedRemoteTs = <lastSavedAt effectif post-fetch>` (remoteTs si remote appliqué, lastSavedAt local sinon, 0 si fetchGist a échoué).
And `_localDirtySinceTs = null`.

### AC12 — Aucune régression sur `connectGist()`
Given un opt-in initial via la modale Settings
When `connectGist()` complète son flow (confirm local-vs-remote ou push initial)
Then le state final (local ou remote) est appliqué selon le flow existant.
And `_lastSyncedRemoteTs = <lastSavedAt du state final>`, `_localDirtySinceTs = null`.

Out of scope:
- Vraie résolution de conflit field-by-field (CRDT, OT). Trop coûteux pour un mono-utilisateur.
- Verrouillage distant (Gist API ne le supporte pas).
- Pull périodique automatique (Q1 = A seul, pas A+B).
- "Voir diff" granulaire item-par-item complet (compteurs + top 10 items à titre divergent uniquement, pas de hash, pas de `lastModifiedAt`).
- Couverture du cas "session périmée fermée sans conflit détecté en cours" : reste un risque résiduel assumé (impossible à couvrir sans GET sync en unload).
- Audit Trail inter-session.
- Indicateur visuel persistant "modifs locales non pushées en attente" hors toasts.

Assumptions:
- `fetchGist()` ([`index.html:2679`](index.html#L2679)) est sûr et idempotent (juste un GET).
- L'introduction des 4 variables module-level (`_lastSyncedRemoteTs`, `_localDirtySinceTs`, `_syncConflict`, `_syncCheckInFlight`) suffit pour porter toute la logique sans toucher au schéma de données.
- L'event `visibilitychange` est plus fiable que `focus` sur Chrome / Safari modernes ; on écoute les deux pour redondance, déduplication via `_syncCheckInFlight`.
- `showToast()` ([`index.html:3043`](index.html#L3043)) est conservé pour les toasts volatils. Le toast conflit utilise un container DOM **distinct** (`#syncConflictToast`) pour ne pas casser le timing 3s ni se faire écraser.
- Les modales utilisent le pattern existant (cf. `#itemModal`, `#confirmModal`).
- Le timestamp réellement écrit dans les payloads Gist est capturé en variable locale (`tsPushed`) et réutilisé pour mettre à jour `_lastSyncedRemoteTs` — pas de `Date.now()` recalculé après l'appel pour éviter les divergences entre payload et snapshot.

Risks:
- **R-A** Race conditions push-guard vs 3 instances : entre le GET (push-guard) et le PATCH, une 3ᵉ instance peut pusher. Mitigation : cas extrême, `_lastSyncedRemoteTs` minimise mais ne supprime pas. Documenté comme limite acceptée.
- **R-B** Surcoût rate limit GitHub : push-guard = 1 GET supplémentaire par push. Avec debounce 2,5s, max ~24 push/min — largement sous 5000/h. Pas un risque réel.
- **R-C** Bug subtil sur l'état sync : 4 variables liées (sync, dirty, conflict, in-flight) doivent rester cohérentes. Mitigation : helper unique `applyRemoteState` + transitions documentées dans chaque AC + tests E2E ciblés AC1/AC2/AC3.

## PATCH_NOTES

<!-- FR markdown body for GitHub Release -->

### Sync multi-instances — fini l'écrasement silencieux entre 2 sessions ouvertes

Quand Atelier est ouvert sur deux machines (Mac + Windows) ou deux onglets en parallèle, la machine qui pushait en dernier écrasait silencieusement les modifs de l'autre. Cette release ajoute trois garde-fous coordonnés pour protéger les données et signaler les conflits sans bloquer le flow.

#### Pull au focus / retour de visibilité

Quand l'onglet ou la fenêtre redevient visible, l'app refetch le Gist en arrière-plan et compare avec sa baseline locale.

- Si le distant est plus récent **et** aucune modif locale n'est en attente → application silencieuse + petit toast « ↺ Données mises à jour (depuis une autre machine) ».
- Si le distant est plus récent **et** des modifs locales ne sont pas encore poussées → toast conflit (voir ci-dessous), aucune écriture n'écrase l'autre.
- Pattern Slack/Notion, économe en rate limit GitHub : pas de polling périodique, juste un GET au réveil.

#### Push-guard avant chaque push

Avant chaque PATCH du Gist (debounce 2,5 s), un GET préalable compare le `lastSavedAt` distant à la baseline locale.

- Distant inchangé → push normal, baseline réalignée.
- Distant plus récent → push **bloqué**, toast conflit affiché, données locales préservées.
- Erreur réseau sur le GET → push annulé avec toast « ⚠ Sync indisponible — réessaie plus tard. ». Les modifs locales restent en IndexedDB et seront retentées à la prochaine modif.

#### Toast conflit avec 3 actions

Quand un conflit est détecté, un toast persistant s'affiche avec :

- **Recharger** → applique la version distante, abandonne les modifs locales en mémoire.
- **Garder local** → push forcé qui écrase la version distante (sans push-guard pour ne pas reboucler sur le conflit).
- **Voir diff** → ouvre une modale avec un résumé comparatif (clients / items / sprints / epics, deltas en + / - et liste des items à titre divergent, top 10).

Le toast reste visible tant que l'utilisateur n'a pas tranché ; aucun nouveau push n'est tenté pendant ce temps.

#### Garde-fous bord à bord

- **`beforeunload`** : le push opportuniste à la fermeture est inhibé si un conflit est ouvert **ou** si un GET sync est en vol — fini le risque d'écraser un remote dont on n'a pas encore vu le verdict.
- **Mode test** (`atelier-mode-test`) : aucun listener `focus`/`visibilitychange` n'est attaché, aucun fetch / push Gist n'est tenté. Cohérent avec le no-op global du mode test.
- **`connectGist()`** (opt-in initial) : flow existant conservé (confirm dialog local-vs-remote), mais la baseline est désormais alignée sur `remote.lastSavedAt` quand on charge le distant pour éviter toute dérive d'horloge.

#### Tests

13 nouveaux tests Playwright dans `tests/sync-multi-instances.spec.js` couvrant les 12 AC du PRD et les 2 cas concurrents identifiés en CODE_REVIEW (focus GET + timer push qui expire ; `beforeunload` pendant un GET en vol). Le mock du Gist est instrumenté via `page.route` pour simuler 2 instances avec un payload partagé.

#### Limites connues (assumées)

- **Rare 3ᵉ instance** : entre le GET du push-guard et le PATCH effectif, une 3ᵉ instance peut encore pusher. Le risque résiduel est documenté ; pour la cible Atelier (mono-utilisateur), 2 instances suffisent largement.
- **Fermeture pendant check remote** : si l'utilisateur ferme l'onglet pile pendant un GET sync, les modifs en cours de debounce (≤ 2,5 s) ne sont pas pushées. Elles restent en IndexedDB local et reviennent au boot suivant — pas de perte de données réelle, juste un délai.
