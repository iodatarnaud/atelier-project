## CADRAGE

Goal:
Introduire un horodatage FR explicite sur chaque WI pour mesurer la durée réelle d'un chantier (ouverture → fermeture). Aujourd'hui le timing n'est trace que dans les commits git (eux-mêmes peu lisibles pour Arnaud), pas dans le WI lui-même qui est pourtant la source de vérité du chantier.

Persona impact:
- Arnaud — visibilité directe sur la durée de chaque chantier (planification, retex, comparaison FAST_TRACK vs STANDARD).
- Claude / Codex — auto-discipline : un timer matérialisé incite à fermer un WI plutôt que le laisser traîner en INDEFINITELY OPEN.

Context:
- Item backlog : ATE-27 « Timer horodaté FR sur chaque WI (ouverture / fermeture / durée) » (méta-protocole).
- État actuel :
  - Aucune section dédiée au timing dans `ai-system/WI_TEMPLATE.md` (vu lors du WI-005).
  - Le header WI contient `Status`, `Owner`, `Branch`, `Target version`, `Mode`, `Risk` — mais pas de date.
  - Le seul horodatage disponible est `git log` sur les commits du WI (feat/bump/merge/CLEANUP/DONE), avec format ISO UTC peu lisible en français.
- Précédent méta-WI : WI-002 (protocole v2.1) — méta-WI exécuté en STANDARD, branche `chore/wi-002-...`, pas de bump app, `Target version: système v...`. WI-006 suit le même pattern mais en FAST_TRACK car le changement est plus restreint (3 fichiers ai-system/ au plus, Risk LOW).

Constraints:
- Pas de bump version `index.html` (méta-WI sur le protocole).
- Pas de modification du contrat de phase (les 13 phases restent les mêmes).
- Compatibilité avec les WI existants (WI-001 à WI-005) : ils restent valides sans la section TIMING — pas de migration rétroactive.
- Pas d'automatisation cron/hook : l'horodatage est rempli par Claude au boot des phases concernées (CADRAGE, DONE).

Out of scope:
- Mesure du temps par phase (`CADRAGE` → `PRD` → `PLAN_REVIEW` → ...) : trop granulaire, pas demandé.
- Calcul de la durée nette (excluant les heures hors travail / week-ends) : la durée brute calendaire suffit pour le besoin Arnaud.
- Tag automatique des WI dépassant un seuil de durée : prématuré, on observe quelques WI avant.
- Modification rétroactive des WI-001 à WI-005 pour ajouter le timing (impossible sans information fiable).

Open questions:

1. **Format horodatage FR** — `JJ/MM/AAAA HH:MM` (granularité minute, timezone Europe/Paris) ou `JJ/MM/AAAA HH:MM:SS` (granularité seconde) ?
   - **(A)** Minute : `01/05/2026 23:00`. Lisible, suffisant pour mesurer en heures/jours.
   - **(B)** Seconde : `01/05/2026 23:00:42`. Plus précis mais bruit visuel.

2. **Trigger d'horodatage** — qui pose les dates et quand ?
   - **(A)** Claude calcule auto via `date +'%d/%m/%Y %H:%M'`. `Opened` au boot CADRAGE (création du WI). `Closed` au passage `Status: DONE`. `Duration` calculé à la fermeture.
   - **(B)** Première transition CADRAGE → autre phase = `Opened`. Reflète mieux "début du travail".
   - **(C)** Manuel par Arnaud dans `CADRAGE.Decisions`. Plus de friction.

3. **Emplacement dans le WI** — où mettre le bloc timing ?
   - **(A)** Bloc dédié `## TIMING` entre le header et `## 0. SESSION CONTROL`. Lisible, pas de pollution du header force-read.
   - **(B)** Champs dans le header (`Opened:`, `Closed:`, `Duration:` directement après `Risk`). Plus compact mais alourdit le `head -8` actuel.

Decisions:
1. **Q1 → (A) Minute.** Justif : `JJ/MM/AAAA HH:MM` lisible, suffisant pour mesurer la durée en heures et jours. La granularité seconde est du bruit pour ce besoin.
2. **Q2 → (A) Auto par Claude.** Justif : zéro friction pour Arnaud, cohérent avec « Claude exécute, Arnaud décide les décisions métier ». Triggers clairs : `Opened` au boot CADRAGE (date du jour), `Closed` au passage `Status: DONE`, `Duration` calculée à la fermeture (format `Xj Yh Zmin` ou `Xh Ymin` selon ordre de grandeur).
3. **Q3 → (A) Bloc `## TIMING`.** Justif : le header `head -8` reste à 8 lignes (force-read existant inchangé). Le bloc `## TIMING` est inséré entre le header et `## 0. SESSION CONTROL`. Lisible visuellement, séparé sémantiquement.

Note canal : décisions transmises par Arnaud via chat (« je valide les 3 »), inscrites par Claude. Pattern récurrent depuis WI-003.

Note méthodo : branche `chore/wi-006-...` car méta-WI sur le protocole (cohérent WI-002). Mode FAST_TRACK car changement restreint (3 fichiers `ai-system/` max, pas de logique métier).
