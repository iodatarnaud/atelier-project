## CADRAGE

Goal:
Ajouter dans la modale détails d'un item Atelier (`index.html`) un champ **Durée réelle (heures)** distinct de l'estimation existante (en jours), permettant à Arnaud de saisir a posteriori le temps réellement passé sur un WI une fois terminé. Calculer et afficher une **agrégation en jours** (au choix : par projet, par sprint, en header global) cohérente avec l'agrégation d'estimation déjà en place. L'objectif retex est de comparer estimé vs réel sur les WI passés et calibrer les estimations futures.

Persona impact:
- **Arnaud** — gain : voir l'écart estimé vs réel par WI et au global, calibrer les estimations futures, capitaliser sur le retex. Coût : un input à saisir à la fin de chaque WI (1-2 secondes).
- Pas d'impact externe (app mono-utilisateur).

Context:
- Le pipe AI capture déjà `Opened/Closed/Duration` dans le header de chaque fichier WI markdown (`work-items/WI-NNN.md`) depuis la v2.2 (cf WI-006). Ce sont des durées **calendaires** (entre `date Opened` et `date Closed`), pas des durées **homme** (temps réellement passé hors interruptions).
- L'app Atelier est aujourd'hui dissociée de ce pipe : elle ne lit pas les fichiers WI markdown. Le champ `actualHours` à ajouter sur l'item est donc **saisi manuellement** par Arnaud (pas pré-rempli auto en MVP).
- Le pattern d'`estimate` existant dans `index.html` :
  - Champ item : `estimate` (number, jours), validé par `_clampNumber(raw.estimate, 0, 1000, 0)` ligne 2588.
  - Tracking : `TRACKED_FIELDS` set inclut `estimate` ligne 2448 (changements horodatés en activité).
  - Badge UI : classe `.est-badge` affiche `${item.estimate || 0}` ligne 3773.
  - Saisie modale : label "Estimation (jours)" ligne 4820.
  - Agrégation : `c.items.filter(i => i.status !== 'done').reduce((a, i) => a + (parseFloat(i.estimate) || 0), 0)` (somme **restant à faire**) lignes 3460/3563.
- Le nouveau champ `actualHours` doit suivre le même pattern, avec deux différences clés :
  - Unité **heures** côté saisie (input number, décimal acceptable type `1.5`), agrégation **jours** (1 jour = ?h, à figer).
  - Sémantique d'agrégation **inverse** : on veut probablement la somme **terminée** (`status === 'done'`) puisque le réel n'a de sens qu'après coup, pas le restant.

Constraints:
- Ne pas casser la compatibilité Gist (`atelier-data.json`) : nouveau champ optionnel, items existants (sans `actualHours`) doivent continuer à s'afficher correctement.
- Ne pas casser les tests E2E Playwright existants. Ajouter au moins 1-2 tests sur la saisie + agrégation.
- Pattern `escapeHtml` / `escapeAttr` conservé sur tout rendu dynamique (cf v0.17.1).
- Mode test (`atelier-mode-test`) : seed démo doit fournir un échantillon avec `actualHours` non nul pour que le rendu agrégation soit testable visuellement.
- Mutation pattern obligatoire : `state mutation → saveState() → render()`.

Out of scope:
- Pré-remplissage auto depuis le fichier `work-items/WI-NNN.md` correspondant (parsing `## TIMING.Duration` du markdown). Hors-scope MVP : nécessiterait fetch raw GitHub API + lien explicite ATE-N ↔ WI-NNN. Possible WI futur si l'usage manuel s'avère pénible.
- Suivi micro de durée (timer in-app, démarrer/arrêter chrono pendant le travail). Hors-besoin actuel.
- Intégration avec un outil externe (Toggl, Harvest, etc.). Pas demandé.
- Saisie en h+min "1h30" si on prend la voie décimal "1.5". Cf Open questions Q2.

Open questions (à trancher en `Decisions` par Arnaud, ou par défaut Claude propose recommandations) :

1. **Conversion h → j** : convention combien d'heures = 1 jour ouvré ? Recommandation Claude : **8h = 1j** (convention standard conseil/dev).
2. **Format saisie** : input décimal (`1.5`, `0.25`) OU h+min (`1h30`, `0h15`) ? Recommandation Claude : **décimal** — plus simple côté input HTML (`type=number step=0.25`), conversion display côté UI si besoin (`1.5h` → "1h30" en lecture).
3. **Quand saisir** : champ accessible librement à tout statut OU verrouillé jusqu'à `status === 'done'` ? Recommandation Claude : **librement** mais avec **affichage discret** tant que statut ≠ `done` (placeholder "à saisir à la fin du WI"), agrégation prend en compte uniquement les items `done` quoi qu'il arrive.
4. **Pré-remplissage auto depuis fichier WI markdown** : MVP fait ça ou non ? Recommandation Claude : **non MVP** (cf Out of scope). Saisie 100% manuelle.
5. **Affichage agrégation** : (A) total séparé `X.Yj réels` à côté de `X.Yj à faire` dans le header global ? (B) badge écart par item (vert si réel < estimé, rouge sinon) ? (C) agrégation comparative `X.Yj réels / Z.Wj estimés` dans header + sidebar projet ? Recommandation Claude : **(C)** dans header global (plus informatif, retex direct) + **(B)** badge discret sur les items `done` (sans surcharge UI).

Decisions:
1. **Conversion h → j** : **8h = 1j** (convention standard).
2. **Format saisie** : **(A) décimal** — input HTML `type=number step=0.25` (tape `1.5` pour 1h30, `0.25` pour 15min).
3. **Quand saisir** : **(A) librement** à tout statut, mais agrégation prend **uniquement les items `status === 'done'`** en compte. Affichage placeholder discret tant que statut ≠ done.
4. **Pré-remplissage auto** depuis `work-items/WI-NNN.md` : **(A) non, saisie 100% manuelle** — la `Duration` calendaire du fichier WI ≠ temps homme réel, pré-remplir avec une donnée trompeuse vaut moins que rien.
5. **Affichage agrégation** : **(A) seulement** — total séparé `X.Yj réels` à côté de `X.Yj à faire` dans le header global. Pas de badge écart par item, pas d'agrégation comparative fusionnée. Minimaliste.

Note canal : Q1-Q5 transmises par Arnaud via chat (Claude pose Q1 → Arnaud répond → Claude pose Q2 → ... → Q5), inscrites par Claude. Pattern récurrent depuis WI-003.

## ✨ Nouveautés

### Saisie de la durée réelle des WI (ATE-28)

Vous pouvez désormais **saisir le temps réellement passé** sur un item (en heures) via la modale détails — distinct de l'estimation existante (en jours) qui reste utile pour le retex calibration.

- Nouveau champ **« Durée réelle (heures) »** dans la modale détails de chaque item, juste après « Estimation (jours) ». Saisie décimale au pas de 0.25 (`0.25` = 15 min, `1.5` = 1h30, `8` = 1j ouvré).
- Saisie **libre à tout statut** ; un placeholder discret « à saisir à la fin du WI » rappelle l'usage tant que l'item n'est pas terminé.
- L'agrégation prend en compte **uniquement les items terminés** pour cohérence retex.

### Agrégation header global — comparaison réalisés / estimés

Le header global affiche désormais une **comparaison cohérente** sur les items terminés :

```
19 items · 21.75 j à faire · 5.38 j réalisés / 7 j estimés (terminés)
```

- **Périmètre commun** : la comparaison réalisés vs estimés porte exclusivement sur les items `status === done` — plus de cross-set incohérent (estimé non-done vs réel done).
- **Conversion 1 jour = 8h** (convention conseil/dev) pour passer des heures saisies aux jours d'agrégation.
- **Header propre** par défaut : la section « réalisés / estimés (terminés) » n'apparaît qu'à partir du moment où au moins un item terminé a un `actualHours` ou un `estimate` non nul.

## 🛠️ Détails techniques

- Nouveau champ optionnel `actualHours` sur les items, validé par `_clampNumber 0..10000`. **Rétro-compatible** : les items existants sans `actualHours` continuent de fonctionner et défaut à 0 lors du `_normalizeItem`.
- Champ tracké dans la timeline d'activité (`TRACKED_FIELDS`) : les modifications de durée réelle apparaissent dans l'historique de l'item (label « Durée réelle », formaté en `Xh`).
- Seed démo enrichi : 4 items terminés ont une `actualHours` non nulle (14h, 6h, 18h, 5h) pour illustrer l'agrégation en mode test.

## 🧪 Tests

- 3 nouveaux tests E2E Playwright (`tests/duree-reelle.spec.js`) : saisie + agrégation done-only + filtre exclusion non-done + header propre par défaut.
- Suite complète : **146/146 PASS**.
