## CADRAGE

Goal:
Appliquer les fixes identifiés en rétrospective WI-001 sur `ai-system/00_AI_SYSTEM.md` et/ou `ai-system/WI_TEMPLATE.md` pour réduire les frictions observées lors de la première utilisation complète du protocole.

Persona impact:
- Claude / Codex / Arnaud auront des règles plus claires sur qui transitionne le Status quand
- Moins d'improvisation au moment des phases (notamment MERGE_RELEASE, VALIDATION_UI, fix de tests)
- Le système est plus déterministe et moins sujet à interprétation

Context:
WI-001 (drag & drop dueDate calendrier) vient d'être livré DONE en suivant l'AI Orchestration Protocol v2 de bout en bout (12 phases, 118 tests verts, v0.15.0 en prod). Ce WI a été le premier "vrai" test du protocole. 6 frictions concrètes ont été observées et listées en rétro. Ce WI-002 les fixe.

Constraints:
- Pas de modif au code de l'app (`index.html`, `tests/`) — méta-WI sur les fichiers méthodo uniquement
- Préserver la compatibilité avec WI-001 (déjà DONE) — pas de re-rédaction rétroactive
- Garder le système strict et minimal : pas d'ajout de complexité non justifiée par une friction observée
- Phases TESTS / VALIDATION_UI / MERGE_RELEASE adaptées : pas de tests E2E (pas de code app), pas de bump version, pas de release GH (juste merge + push)

Out of scope:
- Refonte complète du protocole
- Ajout de phases non observées comme manquantes (ex: pas de "PLANNING" séparé de PRD)
- Modifications de `AI.md`, `CLAUDE_BOOTSTRAP.md`, `CODEX_BOOTSTRAP.md`, `SYSTEM_USAGE.md` sauf nécessité directe d'un fix
- Tooling automation (ex: scripts qui transitionnent le Status auto)

Open questions:

1. **Transitions Status explicites** — friction #1 : "qui transitionne quand" pas clair. Tu as dû manuellement set/revert le Status plusieurs fois.
   - Fix proposé : règle dans `00_AI_SYSTEM.md` "l'agent qui finit sa phase MUST set `Status` au prochain phase ET `Owner` au prochain owner, dans le même output que la section écrite. Pas d'autre agent ne touche au Status."
   - Valider ce fix ? Amender ?

2. **Canal "next" pour Arnaud (VALIDATION_UI)** — friction #2 : Arnaud n'a pas de mécanisme propre pour valider VALIDATION_UI puis transitionner.
   - Fix proposé : convention "Arnaud édite la section VALIDATION_UI à la main dans son IDE (Verdict: OK + cases), puis dit `next` à Claude qui set Status → MERGE_RELEASE."
   - Valider ? Ou autre approche (Arnaud transitionne lui-même via IDE) ?

3. **Ordre MERGE_RELEASE / PATCH_NOTES** — friction #3 : circularité, la release GH a besoin des notes mais PATCH_NOTES vient après.
   - Fix proposé : inverser l'ordre dans la transition table → `CODE_REVIEW → VALIDATION_UI → PATCH_NOTES → MERGE_RELEASE → DOCS → CLEANUP`. Comme ça MERGE_RELEASE peut faire `gh release create --notes-file work-items/WI-XXX.md` (extraire la section).
   - Valider ?

4. **Fix de tests vs fix d'IMPL** — friction #4 : quand 4 tests ont échoué, j'ai modifié les tests (helper test-side), pas le code app. Le contract dit `TESTS (red) → IMPLEMENTATION` sans nuancer.
   - Fix proposé : préciser dans `00_AI_SYSTEM.md` "TESTS (red) → IMPLEMENTATION SAUF si la cause est purement test-side (test mal écrit, sélecteur fragile, helper Playwright instable). Dans ce cas, loop interne TESTS jusqu'à vert."
   - Valider ? Ou tracker quand même comme un retour IMPL pour traçabilité ?

5. **Modifs hors WI en parallèle** — friction #5 : tu as enrichi `ai-system/` pendant WI-001. J'ai improvisé un commit séparé.
   - Fix proposé : règle "Modifs hors WI (ai-system/, docs/, autres) doivent être committées séparément AVANT le commit principal du WI, pour rester clean dans l'historique git."
   - Valider ?

6. **Section 0. SESSION CONTROL** — friction #6 : reste cochée des phases précédentes, pas clair si état persistant ou checklist par-exécution.
   - Fix proposé : règle "à chaque transition de phase, l'agent qui prend la main RESET les flags 0. SESSION CONTROL à NO/NO/... puis les recoche au boot. Sert de marqueur explicite que le boot a bien eu lieu pour cette phase."
   - Valider ?

Decisions:

1. **Transitions Status explicites** (validé) — L'agent qui finit sa phase MUST set `Status` au prochain phase ET `Owner` au prochain owner, dans le même output que la section écrite. **Cas spécial Arnaud** : il ne touche jamais au `Status` lui-même. Il édite uniquement la section dont il est owner (`CADRAGE.Decisions` ou `VALIDATION_UI`) puis dit `next` à Claude qui lit la section, valide qu'elle est complète, et fait la transition selon la table.

2. **Canal `next` pour Arnaud (VALIDATION_UI)** (validé) — Arnaud teste l'UI en Live Server, puis édite la section `VALIDATION_UI` du WI dans son IDE (au minimum `Verdict: OK | NOT OK` + `Issues:`). Quand fait, il dit `next` à Claude qui lit la section et transitionne. Si Arnaud dit juste `next` sans avoir édité la section, Claude bloque avec un BLOCKER "VALIDATION_UI section vide".

3. **Ordre PATCH_NOTES / MERGE_RELEASE inversé** (validé) — Nouvelle transition table : `CODE_REVIEW → VALIDATION_UI → PATCH_NOTES → MERGE_RELEASE → DOCS → CLEANUP`. PATCH_NOTES rédigé AVANT MERGE_RELEASE. La phase MERGE_RELEASE utilise la section `PATCH_NOTES` comme source de vérité unique pour `gh release create --notes-file` (extraction de la section ou via fichier temp).

4. **Fix de tests vs fix d'IMPL** (validé) — Si `TESTS` retourne red :
   - Si la cause est dans le code app (bug d'impl, AC non couvert, edge case raté) → transition `TESTS → IMPLEMENTATION`.
   - Si la cause est purement test-side (test mal écrit, sélecteur fragile, helper Playwright instable, race condition côté test) → loop interne dans `TESTS` jusqu'à vert. Documenter le pourquoi dans la section `TESTS.Notes` (ex: "AC3 refactoré pour `simulateDnd` car `dragTo` inconsistant sur cibles avec enfants").

5. **Modifs hors-périmètre = anomalie BLOCKER** (validé) — Pendant l'exécution d'un WI, AUCUN acteur (Claude, Codex, Arnaud) ne fait de modifs hors-périmètre du WI courant. Toute modif système / méthodo / `ai-system/` / `docs/` passe par un méta-WI dédié (style WI-002). Pendant `MERGE_RELEASE`, Claude vérifie `git diff` :
   - Si tout est in-scope → commit feature + commit bump → merge normalement.
   - Si modifs hors-périmètre détectées → BLOCKER, stop, demander instruction (commit séparé ad hoc OU stash + traiter dans un autre WI).

6. **`0. SESSION CONTROL` reset par-phase** (validé) — À chaque transition de phase, l'agent qui prend la main RESET tous les flags `0. SESSION CONTROL` à NO en début d'exécution, puis les recoche à YES au fur et à mesure du boot reload (lecture AI.md, 00_AI_SYSTEM, WI, fichiers source, git diff). Sert de marqueur explicite que le boot a bien eu lieu pour CETTE phase, pas une trace cumulative.

## PRD

Objective:
Appliquer les 6 fixes validés en CADRAGE pour réduire les frictions observées sur le AI Orchestration Protocol v2 (rétro WI-001). Mise à jour des fichiers `ai-system/` uniquement, aucun impact code app.

Current behavior:
Le protocole v2 actuel décrit dans `ai-system/00_AI_SYSTEM.md` ne précise pas explicitement : (1) qui transitionne le Status quand, (2) comment Arnaud signale la fin de sa phase, (3) la circularité PATCH_NOTES/MERGE_RELEASE pour les release notes, (4) la distinction fix-tests vs fix-IMPL, (5) le traitement des modifs hors-périmètre, (6) le cycle de vie des flags `0. SESSION CONTROL`.

Target behavior:
`ai-system/00_AI_SYSTEM.md` (et secondairement `WI_TEMPLATE.md` + `SYSTEM_USAGE.md`) intègrent 6 règles formelles (cf. AC1-AC6) qui rendent le protocole déterministe sur les 6 frictions identifiées.

User stories:
- US1 — En tant que Claude/Codex, je sais sans ambiguïté qui set le Status au prochain phase à la fin d'une exécution.
- US2 — En tant qu'Arnaud, j'ai un canal explicite pour valider une phase qui m'appartient (édite section + `next`).
- US3 — En tant que Claude, je peux créer la release GH avec les notes définitives sans avoir à dupliquer le changelog.
- US4 — En tant que Claude/Codex, je sais distinguer un échec de tests "côté app" d'un échec "côté tests" et router vers la bonne phase.
- US5 — En tant que système, j'ai zéro tolérance pour les modifs hors-périmètre pendant un WI (BLOCKER explicite).
- US6 — En tant que tout agent, je vois clairement si le boot a bien été refait pour CETTE phase (pas un état hérité).

Acceptance criteria:

### AC7 — Table canonique des phases dans `00_AI_SYSTEM.md`
Given le fichier `ai-system/00_AI_SYSTEM.md`

### AC11 — Résolution du WI courant
