## CADRAGE

Goal:
Permettre à l'utilisateur de modifier la `dueDate` d'un item en le draggant entre cellules du calendrier (replanifier), depuis la section "Non planifiés" vers une cellule (planifier), ou vers la section "Non planifiés" (dé-planifier). Sans ouvrir la modale détail.

Persona impact:
Le consultant Salesforce replanifie sa charge en un seul geste depuis la vue Calendrier — friction divisée par N par rapport au flow actuel (ouvrir modale → changer date → enregistrer).

Context:
Livraison 2 d'ATE-6. La livraison 1 (vue Calendrier multi-projets, v0.14.4) a été cadrée et livrée sans D&D, le D&D étant explicitement marqué "out of scope, livraison future". Les items "Non planifiés" sont stackés en bas de la vue ; Arnaud demande explicitement de pouvoir les dragger vers une cellule.

Constraints:
- Préserver `withItemChangeTracking()` pour générer un event `change` sur le champ `dueDate` (cohérence avec l'Activité du WI, ATE-7+8)
- Ne pas casser les filtres calendrier (projet, type, prio, epic), l'expand/collapse de cellules, la section "En retard"
- Ne pas casser les D&D existants (board, sidebar→sprint, sidebar→epic, backlog groupé)
- Pas de framework, pas de lib externe : HTML5 drag & drop natif comme partout dans l'app
- Le calendrier est multi-projets : un D&D ne change PAS le projet de l'item, uniquement sa `dueDate`

Out of scope:
- Reorder intra-cellule (pas d'ordre métier intra-jour)
- D&D depuis le backlog vers le calendrier ou inverse
- D&D multi-sélection
- Snap intelligent (ex: éviter les week-ends)

## PRD

Objective:
Permettre à l'utilisateur de modifier la `dueDate` d'un item depuis la vue Calendrier en le déplaçant à la souris (HTML5 D&D natif), sans ouvrir la modale détail. Aucune régression sur les D&D existants (board, sidebar, backlog).

Current behavior:
Vue Calendrier (v0.14.4) : grid mensuelle multi-projets, items avec dueDate dans les cellules, items sans dueDate dans la section "Non planifiés", items overdue+non-done dans la section "En retard" (peuvent apparaître 2 fois si dans le mois courant ET overdue). Click item ouvre la modale détail. Aucun D&D dans la vue Calendrier.

Target behavior:

Tous les `.cal-item` des cellules + section "Non planifiés" + section "En retard" sont `draggable="true"` et portent les attributs stables `data-client-id`, `data-item-id`. Toutes les `.cal-cell` portent `data-date`.

Le drag transporte un payload calendrier dédié via une **variable de module** `calendarDragItem = { clientId, itemId }` (plus `dataTransfer.setData('atelier/calendar', JSON.stringify(payload))` pour redondance). **Ne jamais réutiliser** `dragItemId` global (réservé board/sidebar/backlog) afin d'éviter toute interférence avec les handlers existants.

Filtrage explicite côté handlers D&D existants : `attachDragDrop` (board), `attachSidebarDropTargets`, `attachBacklogDragDrop` doivent ignorer le drop si `calendarDragItem !== null` (drag d'origine calendrier). Le drag calendrier ne doit pas non plus déclencher leurs handlers de drop.

Cibles de drop :
- `.cal-cell` avec `inMonth: true` (mois courant) → `dueDate ← cell.date` formaté `YYYY-MM-DD` via `localDateKey()`
- `.calendar-unplanned-items` (section "Non planifiés") → `dueDate ← null`
- `.cal-cell.other-month` → drop **ignoré** (event preventDefault sur dragover refusé, pas de `.drag-over`, pas de mutation)
- `.calendar-overdue-items` (section "En retard") → drop **ignoré** (vue dérivée)
- Hors zone → drop natif sans handler, pas de modif

Mutation guard (no-op) :
Avant d'appeler `withItemChangeTracking(item, () => { item.dueDate = newValue })`, comparer `item.dueDate === newValue` (avec `null` géré). Si égal → ne pas appeler `withItemChangeTracking`, ne pas `saveState()`, ne pas `renderView()`. Évite les events parasites et les rerenders inutiles.

Tracking : la mutation passe par `withItemChangeTracking()`. `dueDate` est dans `TRACKED_FIELDS` (ATE-7+8) donc un event `{ type: 'change', field: 'dueDate', before, after }` est généré dans `item.activity`. Puis `saveState()` puis `renderView()`.

Anti-click post-drag :
Pattern `dragJustHappened` (ou équivalent : `lastDragEndAt = Date.now()` à `dragend/drop`, et dans `openItemFromCalendar` early-return si `Date.now() - lastDragEndAt < 250ms`). Sinon, après un drop, le click capturé sur le `.cal-item` ouvrirait la modale détail sans intention.

Feedback visuel :
- Item draggué : classe `.dragging` (opacité réduite, pattern existant board/sidebar)
- Cible valide survolée : classe `.drag-over` sur la `.cal-cell` ou `.calendar-unplanned`
- Cibles invalides (other-month, En retard) : **jamais** de `.drag-over`. Curseur `default` (HTML5 ne pas appeler `e.preventDefault()` sur dragover → drop natif rejeté).

Attachement listeners :
Dans `renderCalendar()`, après `container.innerHTML = ...`, appeler `setTimeout(attachCalendarDragDrop, 0)` (pattern board). `attachCalendarDragDrop()` scope ses queries sous `.calendar-view` et attache `dragstart` sur tous les `.cal-item`, `dragover`/`dragleave`/`drop` sur tous les `.cal-cell:not(.other-month)` + la section `.calendar-unplanned`, `dragend` sur tous les `.cal-item` (cleanup).

Le projet (`item.clientId`) n'est jamais modifié par un D&D calendrier, même en mode "Tous projets" et même si l'item appartient à un projet différent du projet actif (`state.activeClientId`). La mutation cible le bon item via `state.clients.find(c => c.id === payload.clientId).items.find(i => i.id === payload.itemId)`.

User stories:
- US1 — Glisser un item d'une cellule vers une autre pour replanifier sans modale.
- US2 — Glisser un item "Non planifié" vers une cellule pour planifier en 1 geste.
- US3 — Glisser un item d'une cellule vers "Non planifiés" pour dé-planifier.
- US4 — Glisser un item de "En retard" vers une cellule future pour replanifier vite.
- US5 — Le tracking d'activité reflète chaque change de dueDate (cohérent avec ATE-7+8).

Acceptance criteria:

### AC1 — D&D entre cellules du mois courant
Given un item I avec `dueDate = 2026-04-15`, affiché dans la cellule `[data-date="2026-04-15"]` (mois courant)
When je drag I et le drop sur la cellule `[data-date="2026-04-22"]`
Then `I.dueDate === '2026-04-22'`, I disparaît de la cellule du 15 et apparaît dans celle du 22.
And `I.activity` contient un nouvel event `{ type: 'change', field: 'dueDate', before: '2026-04-15', after: '2026-04-22', at: <epoch_ms>, id: <activity_id> }`.
And `saveState()` est appelé exactement 1 fois.

### AC2 — D&D depuis "Non planifiés" vers cellule
Given un item I sans `dueDate`, dans `.calendar-unplanned-items`
When je drag I et le drop sur la cellule `[data-date="2026-05-10"]`
Then `I.dueDate === '2026-05-10'`, I disparaît de "Non planifiés" et apparaît dans la cellule du 10.
And `I.activity` contient un event `change` avec `field: 'dueDate', before: null, after: '2026-05-10'`.

### AC3 — D&D vers "Non planifiés" (dé-planifier)
Given un item I avec `dueDate = '2026-04-20'`
When je drag I et le drop sur `.calendar-unplanned`
Then `I.dueDate === null`, I disparaît de la cellule, apparaît dans "Non planifiés".
And `I.activity` contient un event `change` avec `field: 'dueDate', before: '2026-04-20', after: null`.

### AC4 — D&D depuis "En retard" vers cellule
Given un item I overdue (`dueDate = '2026-04-10'`, `status !== 'done'`), affiché dans `.calendar-overdue-items`
When je drag I et le drop sur la cellule `[data-date="2026-04-25"]`
Then `I.dueDate === '2026-04-25'`, I disparaît de "En retard" (n'est plus overdue car >= today) et apparaît dans la cellule du 25.
And `I.activity` contient un event `change` correspondant.

### AC5 — Drop ignoré sur cellule "other-month"
Given un item I quelconque
When je drag I et le drop sur une cellule `.cal-cell.other-month`
Then `I.dueDate` ne change pas.
And `I.activity.length` ne change pas.
And aucun rerender déclenché par mutation.

### AC6 — Drop ignoré sur section "En retard"
Given un item I quelconque
When je drag I et le drop sur `.calendar-overdue-items`
Then `I.dueDate` ne change pas.
And `I.activity.length` ne change pas.

### AC7 — Drop hors zone valide
Given un item I quelconque
When je drag I et relâche en dehors de toute cible (header, padding, sidebar, body)
Then `I.dueDate` ne change pas.
And `I.activity.length` ne change pas.

### AC8 — Feedback visuel pendant le drag
Given un drag en cours sur un item I
When la souris survole une cible valide (cellule mois courant OU `.calendar-unplanned`)
Then la cible a la classe `.drag-over` (cohérent board/sidebar).
And l'item draggué a la classe `.dragging`.
When la souris survole une cible invalide (`.cal-cell.other-month`, `.calendar-overdue`)
Then ni la cible ni les autres cibles n'ont `.drag-over` (jamais de highlight sur cible invalide).

### AC9 — D&D inter-projets en mode "Tous projets" (item d'un projet non actif)
Given `state.calendarFilterProject = null`, `state.activeClientId = ID_projet_B`, et un item I appartenant au projet A (différent de B), visible dans le calendrier
When je drag I et le drop sur une cellule
Then `I.clientId` reste === `ID_projet_A` (le projet n'est PAS modifié), `I.dueDate` change vers la date de la cellule.
And la mutation est bien faite sur l'item du projet A (pas un faux-friend du projet B).

### AC10 — Non-régression sur les D&D existants
Given les D&D existants (board kanban / sidebar sprint+epic / backlog groupé)
When le D&D calendrier est ajouté et un drag calendrier est en cours
Then les autres handlers ignorent le drop calendrier (pas de mutation cross-feature).
And la suite Playwright existante reste verte (88+ tests passent).

### AC11 — Anti-click post-drag
Given un item I dans une cellule du mois courant
When je drag I et le drop sur une autre cellule du mois courant
Then la modale détail (`#itemModal`) **ne s'ouvre PAS** (le click capturé post-dragend est ignoré via `dragJustHappened`).
And la mutation de `dueDate` se fait normalement.

### AC12 — No-op si dueDate inchangée
Given un item I avec `dueDate = '2026-04-15'`
When je drag I et le drop sur la même cellule `[data-date="2026-04-15"]`
Then `I.dueDate` ne change pas, `I.activity.length` ne change pas, `saveState()` n'est pas appelé.
Given un item I sans `dueDate`
When je drag I et le drop sur `.calendar-unplanned`
Then `I.dueDate` reste `null`, `I.activity.length` ne change pas, `saveState()` n'est pas appelé.

### AC13 — Item overdue draggable depuis ses 2 occurrences
Given un item I overdue avec `dueDate` dans le mois affiché ET dans le passé (donc visible 2 fois : section "En retard" + cellule du jour passé du mois courant)
When je drag I depuis l'une OU l'autre des 2 occurrences vers une cellule cible
Then la mutation est appliquée correctement à la même instance d'item (1 seul event `change` ajouté, pas 2).

Out of scope:
- Reorder intra-cellule (pas d'ordre métier intra-jour)
- D&D depuis ou vers le backlog
- D&D multi-sélection
- Snap intelligent (week-ends / jours fériés)
- Accessibilité clavier (mouse-only en v1)
- Modification de tout champ autre que `dueDate` via D&D calendrier
- Highlight visuel des cibles invalides (curseur `default` natif suffit)

Assumptions:
- Variable de module dédiée `calendarDragItem = { clientId, itemId } | null`. Set au `dragstart`, reset au `dragend`. JAMAIS de réutilisation de la variable globale `dragItemId` (réservée board/sidebar/backlog).
- Les handlers D&D existants (`attachDragDrop`, `attachSidebarDropTargets`, `attachBacklogDragDrop`) ajoutent une garde `if (calendarDragItem) return;` en début de leur callback `drop`.
- Pattern `dragJustHappened` via `lastCalendarDragEndAt` (epoch ms). `openItemFromCalendar` early-return si `Date.now() - lastCalendarDragEndAt < 250`.
- `withItemChangeTracking()` + `dueDate ∈ TRACKED_FIELDS` (ATE-7+8) garantissent l'event `change` automatique.
- Listeners attachés via `setTimeout(attachCalendarDragDrop, 0)` après chaque `renderCalendar()`, scopés sous `.calendar-view`.
- Sélecteurs stables ajoutés sur le markup : `data-date` sur `.cal-cell` (déjà présent), `data-client-id` + `data-item-id` sur `.cal-item` (à ajouter dans `renderCalendarItem`).

Risks:
- **Re-render pendant un drag** : si un autre événement déclenche `renderView()` pendant un drag, l'élément source disparaît, le drag se casse. Cas marginal en usage normal (1 utilisateur, pas de timer auto-render). Pas de mitigation v1.
- **Filtres actifs qui cachent l'item après drop** : si la nouvelle `dueDate` rend l'item invisible (filtre projet exclut son projet, filtre type/prio/epic l'exclut), l'utilisateur perd la trace. À documenter dans SMOKE-TEST.
- **Performance avec beaucoup d'items** : ~5000 chips × 4 listeners (dragstart/dragover/drop/dragend) attachés à chaque render. HTML5 natif léger, mais à surveiller en perf si l'app scale. Pas de mesure en v1.
- **D&D mobile/touch** : non supporté par HTML5 natif sans polyfill. L'app n'a pas de garantie mobile. Pas de régression introduite.
- **Drop sur sidebar pendant drag calendrier** : mitigé par garde `if (calendarDragItem) return;` dans les handlers existants (cf. Assumptions).
- **Item overdue avec 2 occurrences DOM** : le payload `{ clientId, itemId }` garantit que la mutation cible le bon item peu importe l'occurrence draggée (cf. AC13).

## ✨ Drag & drop dueDate dans la vue Calendrier (WI-001 / ATE-20)

Replanifie tes items en un geste : drag depuis une cellule vers une autre, depuis "Non planifiés" vers une date, ou depuis une cellule vers "Non planifiés" pour dé-planifier. Plus besoin d'ouvrir la modale détail.

### Cibles supportées
- **Cellule du mois courant** → la `dueDate` devient la date de la cellule
- **Section "Non planifiés"** → la `dueDate` devient `null` (dé-planifier)
- **Cellule "other-month"**, section "En retard", hors zone → drop ignoré (pas de modif accidentelle)

### Garde-fous techniques
