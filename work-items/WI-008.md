## CADRAGE

Goal:
Permettre à Arnaud de piloter en **temps réel** une session multi-agents (Claude Code + Codex CLI tournant chacun dans leur propre terminal sur le Mac) **depuis son iPhone en mobilité**. Le scope d'interaction mobile est restreint à deux types d'inputs : (1) **valider les "next" / continuations** demandées par les agents, (2) **approuver les autorisations** (permissions de tools, exécutions Bash, edits sensibles). Pas de saisie texte longue, pas de pilotage off-line / async — c'est du **live** sur sessions IDE locales actives.

Persona impact:
- **Arnaud** — gain de mobilité : aujourd'hui il doit être physiquement devant le Mac pour répondre aux prompts des agents. En mobilité (transports, attente, déplacement client), les agents restent bloqués. Avec ce pipe, son iPhone vibre → il valide en 2 taps → l'agent reprend. Multiplication directe du temps utile de session AI.
- **Claude Code / Codex CLI (côté agents)** — pas de changement perçu : ils continuent de tourner en mode interactif local sur leurs terminaux respectifs, leurs prompts sont juste **interceptés et relayés** vers le mobile, leurs réponses **réinjectées** par le canal habituel.

Context:
- Workflow trio actuel (Claude/Codex/Arnaud) suppose Arnaud devant son Mac pour : approuver les plans, autoriser les outils, dire "next" entre phases, transmettre les reviews Codex.
- **Setup réel observé** (précision Arnaud, screenshot WI-008) : Arnaud bosse dans **VS Code** sur son Mac. Les agents Claude Code et Codex tournent **chacun comme une extension VS Code native, dans leur propre panneau latéral** (panneau de droite, onglets `CONVERSATION / CLAUDE CODE / CODEX` visibles côte à côte). Ce ne sont **PAS** des CLI lancées dans des terminaux libres / `tmux` / iTerm. Les prompts d'autorisation et les "next" sont saisis via les UI propres des extensions, pas via stdin terminal.
- Conséquences directes :
  - Toute archi reposant sur `tmux send-keys` ou injection clavier dans un terminal est **disqualifiée d'office**.
  - L'interception des prompts doit passer par : (a) les APIs/hooks exposés par les extensions VS Code Claude Code et Codex (à investiguer), (b) l'API d'extension VS Code générique (`vscode.window.onDidShowMessage`, output channels, custom views), (c) `code tunnel` / VS Code Remote Tunnels pour rendre VS Code accessible via `vscode.dev` sur mobile, (d) une combinaison.

- **Repo cible non figé** : ce WI est un chantier d'**outillage** (infra de pilotage) qui n'est probablement pas du code Atelier (`index.html`). Décision repo (Atelier `tools/pilote/` vs nouveau repo dédié vs autre) repoussée après brainstorm Codex.

### Codex Response

Pas de question bloquante avant arbitrage. Les deux seules vérifications terrain à faire avant PRD sont : (1) test iPhone réel de `code tunnel` sur les panneaux Claude/Codex, (2) test d'un hook Claude `Notification`/`PreToolUse` déclenché depuis la session VS Code native.

#### a) Investigation technique

**Verdict global : la voie "extension custom qui approuve automatiquement les extensions Claude/Codex" n'est pas fondable aujourd'hui sans API privée.** Les surfaces publiques observées permettent d'ouvrir/focus les panneaux et parfois d'observer des événements, pas de répondre proprement aux boutons internes `approve/deny/next`.

1. **Claude Code VS Code**
   - Source locale vérifiée : extension installée `anthropic.claude-code` `2.1.126`, manifest `package.json`.
   - Commandes publiques exposées : `claude-vscode.editor.open`, `openLast`, `primaryEditor.open`, `window.open`, `createWorktree`, `sidebar.open`, `newConversation`, `focus`, `blur`, `logout`, `terminal.open`, `acceptProposedDiff`, `rejectProposedDiff`, `insertAtMention`, `showLogs`, etc.
   - Absence notable : aucune commande publique `approve`, `deny`, `next`, `continue`, `respond`, `permission`.
   - Surface utile officielle : les hooks Claude Code existent côté Claude Code et couvrent `PreToolUse`, `PostToolUse`, `Notification`, `Stop`, `UserPromptSubmit`, etc. La doc Anthropic précise que `Notification` se déclenche quand Claude demande une permission ou attend une input, et que `PreToolUse` peut bloquer/contrôler certains appels. Source : https://docs.anthropic.com/en/docs/claude-code/hooks
   - Verdict : **capture Claude plausible via hooks officiels ; injection UI Claude non supportée publiquement.** Pour Claude, le MVP raisonnable n'est pas "cliquer le bouton VS Code à distance", mais "déporter la décision dans un hook/permission policy contrôlé".

2. **Codex VS Code**
   - Source locale vérifiée : extension installée `openai.chatgpt` `26.429.30905`, manifest `package.json`.
   - Commandes publiques exposées : `chatgpt.openSidebar`, `chatgpt.openCommandMenu`, `chatgpt.newCodexPanel`, `chatgpt.addToThread`, `chatgpt.addFileToThread`, `chatgpt.newChat`, debug NUX, LSP MCP args.
   - Absence notable : aucune commande publique `approve`, `deny`, `next`, `continue`, `respond`, `permission`.
   - Documentation officielle disponible : OpenAI confirme Codex en IDE/VS Code/Cursor/Windsurf, mais ne documente pas une API d'extension tierce pour permissions ou réponses programmatiques. Source : https://help.openai.com/en/articles/11369540
   - Signal non officiel mais important : le repo public OpenAI Codex contient des issues `extension` et des demandes de hook permission ; cela suggère que la surface d'automatisation de l'extension n'est pas stabilisée publiquement.
   - Verdict : **capture/injection Codex VS Code non fiable pour WI-008 si on reste dans l'extension native.** La voie robuste pour Codex est soit passer par Codex CLI avec un wrapper protocolaire, soit accepter Remote Tunnels comme interaction humaine directe.

3. **Injection programmatique**
   - `vscode.commands.executeCommand` fonctionne pour les commandes exposées par VS Code ou par les extensions. Source : https://code.visualstudio.com/api/references/commands
   - VS Code expose les commandes déclarées via `contributes.commands`; les manifests locaux ne déclarent pas d'action d'approbation pour Claude/Codex.
   - `webview.postMessage` ne permet pas à une extension tierce de parler au webview privé d'une autre extension sans API/export explicite.
   - `code --command` n'est pas une base fiable ici : le binaire `code` n'est même pas disponible dans ce shell, et même s'il l'était il ne ferait qu'exécuter des commandes publiques.
   - Verdict : **injection UI inter-extension à exclure du PRD MVP.** Toute solution qui dépend d'un sélecteur DOM, d'un message webview privé ou du focus clavier est un hack fragile.

4. **Remote Tunnels / `vscode.dev`**
   - Microsoft documente Remote Tunnels pour se connecter depuis VS Code desktop ou web à une machine distante avec tunnel actif. Source : https://code.visualstudio.com/docs/remote/tunnels
   - Les extensions web ont des limitations de runtime navigateur si elles doivent s'exécuter dans le web extension host. Source : https://code.visualstudio.com/api/extension-guides/web-extensions
   - Nuance importante : avec Remote Tunnels, une partie extension peut tourner côté serveur distant, donc l'absence de champ `browser` dans les manifests Claude/Codex ne prouve pas à elle seule l'incompatibilité. Le vrai test est empirique : rendu du webview, login, boutons permission, focus, clavier iOS.
   - Verdict : **Remote Tunnels est le meilleur spike immédiat, pas l'architecture produit.** Il peut prouver le besoin en 30 minutes ; il ne donnera probablement pas une UX mobile assez bonne pour devenir le produit final.

5. **Sécurité voie A**
   - Mac -> VPS : connexion sortante uniquement, WebSocket ou SSE long-poll, authentification par device token opaque long terme stocké dans Keychain/macOS + rotation manuelle MVP. mTLS est propre mais trop lourd pour MVP.
   - iPhone -> VPS : PWA avec session courte, WebAuthn/passkey si V1, sinon magic link + refresh token court stocké côté serveur avec révocation.
   - Chaque décision doit être liée à un `eventId`, un `agentId`, un hash du payload, une expiration courte et un statut monotone `pending -> accepted|denied|expired`.
   - Verdict : **pas de canal entrant Mac, pas de token statique dans l'URL, pas d'action réutilisable.** Le VPS est un relais d'événements, pas une autorité aveugle.

6. **PWA Web Push iOS**
   - Apple supporte Web Push pour les web apps ajoutées à l'écran d'accueil depuis iOS/iPadOS 16.4, via Push API, Notifications API et Service Workers. Source : https://developer.apple.com/documentation/usernotifications/sending-web-push-notifications-in-web-apps-and-browsers
   - Safari ne supporte pas les push invisibles : il faut afficher une notification à réception, sinon la permission peut être révoquée.
   - Verdict : **PWA viable pour V1, pas pour MVP jour 1.** Pour MVP, une page PWA ouverte + polling/SSE suffit ; Web Push vient après validation du flux.

#### b) Challenge des voies A et B

**Voie A — extension custom + relais OVH + PWA : rejetée comme MVP si elle vise les extensions natives.**

Angles morts :
- Elle suppose l'existence d'APIs publiques `approve/deny/next` qui ne sont pas visibles dans les manifests actuels.
- Elle mélange deux problèmes : notification mobile et contrôle de permission. Le premier est faisable ; le second est bloqué par la surface privée des extensions.
- Elle sous-estime la maintenance : une update Anthropic/OpenAI peut casser un scraper webview ou un protocole interne sans préavis.
- Elle crée un risque sécu disproportionné si le mobile peut approuver des commandes dont le payload complet n'est pas lisible.

Challenge UX :
- Une PWA à 2 boutons est bonne seulement si le prompt est atomique et lisible. Pour `Bash` ou `Edit`, il faut afficher au minimum commande/fichier, cwd, agent, repo, risque, expiration.
- Le mode "tap rapide" peut devenir dangereux : approuver depuis un métro sans voir les détails revient à recréer `--dangerously-skip-permissions` avec une vibration.
- La file multi-agent doit être explicite : une carte par demande, badge agent/repo, jamais un bouton global `Approve next`.

**Voie B — Remote Tunnels : acceptée comme spike, refusée comme produit.**

Forces :
- Zero-code.
- Sécurité et auth largement déléguées à Microsoft/GitHub.
- Interagit avec les vraies UIs, donc pas de problème d'injection.

Faiblesses :
- UX iPhone probablement médiocre : VS Code web, panneaux latéraux, petits boutons, clavier/focus, scroll horizontal.
- Notifications non maîtrisées : tu dépends du navigateur/VS Code web, pas d'un vrai flux "agent blocked".
- Pas de résumé court, pas de file multi-agent claire, pas d'audit trail propre.
- Si le tunnel marche, il risque de masquer le vrai besoin produit : pas "VS Code sur iPhone", mais "débloquer une décision en 10 secondes".

Verdict produit : **B est un outil de secours et un spike de faisabilité, pas le MVP cible.** Le produit cible doit être une inbox mobile de décisions, pas un IDE comprimé.

#### c) Recommandation d'architecture

**Recommandation : hybride B -> A', avec A' recentrée sur des agents pilotables par protocole, pas par webview privé.**

1. **Spike immédiat : Remote Tunnels**
   - Objectif : vérifier si Arnaud peut réellement valider un prompt Claude/Codex depuis Safari iOS.
   - Durée : 30-60 min.
   - Critère : si le tap moyen dépasse 15 secondes ou demande zoom/pan, B reste fallback seulement.

2. **MVP cible : relais mobile pour surfaces hookables**
   - Claude : utiliser hooks officiels (`Notification`, `PreToolUse`) pour créer des événements mobiles. Pour l'approbation, privilégier un mécanisme hook/policy si possible ; sinon limiter MVP à notification + deep link Remote Tunnel.
   - Codex : ne pas tenter l'extension VS Code native en MVP. Soit basculer Codex en CLI dans un terminal contrôlé/wrappé pour ce cas d'usage, soit considérer Codex extension comme "Remote Tunnel only" jusqu'à API publique.
   - Relais : petit service OVH avec event store, SSE/WebSocket Mac, UI mobile web.

3. **V1 : normaliser un protocole `AgentDecisionRequest`**
   - Champs obligatoires : `eventId`, `agent`, `workspace`, `kind`, `title`, `details`, `risk`, `createdAt`, `expiresAt`, `actions`, `payloadHash`.
   - Actions : `approve_once`, `deny`, `open_remote`, `snooze`.
   - Audit : log append-only local + VPS.

Trade-offs :
- Fragilité technique : faible si hooks/CLI ; élevée si webview privé.
- Latence : très bonne avec WebSocket/SSE ; acceptable avec polling 2-5 s ; Remote Tunnels variable.
- Sécurité : maîtrisable avec relais sortant + action IDs expirables ; mauvaise si boutons génériques.
- Ops : OVH PWA ajoute un service mais reste simple ; mTLS/reporting/HA peuvent attendre.
- Install Arnaud : Remote Tunnels minimal ; A' demande extension/script local + PWA ajoutée écran d'accueil.
- Réversibilité : forte si protocole agent-agnostique ; faible si couplage aux DOMs Claude/Codex.
- Dépendance propriétaire : Remote Tunnels dépend Microsoft ; hooks Claude dépendent Anthropic ; Codex extension privée est le pire point.

#### d) Risques majeurs et mitigations

- **API extension privée / breaking changes** : ne pas dépendre du DOM webview ni d'assets minifiés. Mitigation : utiliser uniquement hooks/commands documentés ; sinon fallback Remote Tunnel.
- **Codex extension non hookable** : risque bloquant pour "Codex natif VS Code". Mitigation : PRD doit prévoir une bifurcation Codex CLI wrapper ou Remote Tunnel pour Codex.
- **Approbation dangereuse depuis mobile** : risque P0 sécurité. Mitigation : afficher payload court mais substantiel, hash, cwd, repo, commande/fichier, expiration ; pas d'approbation groupée en MVP.
- **Multi-session race Claude/Codex** : risque confusion. Mitigation : inbox triée par `createdAt`, cartes par agent, action idempotente, état `already resolved` si l'autre écran a répondu.
- **Latence push** : Web Push iOS peut être retardé. Mitigation MVP : PWA ouverte en foreground + SSE/polling ; Web Push en V1 seulement.
- **Panne VPS** : les agents ne doivent pas être bloqués définitivement par le relais. Mitigation : timeout local, fallback VS Code direct, pas de dépendance au VPS pour refuser/continuer localement.
- **Compromission VPS** : risque d'injection d'approbations. Mitigation : Mac vérifie signature/HMAC, expiration, payloadHash, allowlist agent/workspace ; le VPS ne doit pas pouvoir inventer une demande valide.
- **Focus VS Code requis** : si une action nécessite focus clavier/webview, elle est hors MVP. Mitigation : uniquement protocole/hook ou action manuelle Remote Tunnel.
- **Audit insuffisant** : impossible de comprendre une mauvaise approbation. Mitigation : journal local append-only des prompts, décisions, acteurs, timestamps.

#### e) Recommandation repo cible

**Verdict : nouveau repo dédié `atelier-pilote`.**

Justification :
- Ce n'est pas du code produit Atelier (`index.html`) mais de l'outillage personnel multi-composants : extension/script local, relais VPS, PWA, docs d'installation.
- Le cycle de release, les secrets, le déploiement OVH et les tests ne doivent pas polluer le repo Atelier.
- Le protocole `AgentDecisionRequest` mérite d'être versionné indépendamment pour pouvoir ajouter Claude, Codex CLI, puis un troisième agent.

Structure recommandée :
- `packages/agent-bridge/` : daemon/extension locale ou scripts hooks.
- `packages/relay/` : API OVH + store + auth.
- `packages/mobile/` : PWA.
- `packages/shared/` : types, schémas JSON, HMAC/signing, fixtures.
- `docs/` : runbook Mac, runbook OVH, modèle de menace.

À éviter :
- Hybride "scripts dans Atelier + PWA ailleurs" : dispersion inutile.
- Trois repos séparés : trop lourd avant stabilisation.
- Extension VS Code marketplace comme noyau initial : prématuré tant que l'injection inter-extension n'est pas publique.

#### f) Phasage MVP -> V1 -> V2

**MVP fin de semaine : prouver le déblocage, pas industrialiser.**

Scope strict :
- Spike `code tunnel` sur iPhone : documenter utilisable/non utilisable pour Claude et Codex.
- Un seul agent hookable : Claude prioritaire.
- Un seul type d'event : `Claude waiting / permission requested` via hook `Notification`, ou `PreToolUse` si le payload est exploitable.
- Relais minimal : endpoint OVH ou local d'abord, stockage en mémoire/SQLite, polling mobile 2 s ou SSE.
- UI mobile : page web responsive déjà ouverte, cartes de décision, boutons `Ouvrir VS Code` / `Vu` / éventuellement `Approve` seulement si l'action est techniquement supportée sans hack.
- Sécurité MVP : token device unique, HTTPS, eventId expiré, pas de push, pas de HA, pas de rotation automatique.

Critère de succès MVP :
- Depuis l'iPhone, Arnaud sait qu'un agent est bloqué en moins de 5 secondes quand la PWA est ouverte.
- Il peut comprendre quelle décision est attendue sans zoomer VS Code.
- Il peut débloquer au moins Claude soit par action mobile supportée, soit par deep link/fallback Remote Tunnel clairement plus rapide que retourner au Mac.

**V1 : produit utilisable au quotidien.**

- PWA installée écran d'accueil + Web Push iOS.
- Multi-agent au niveau protocole : Claude + Codex CLI wrapper ou Codex extension fallback tunnel.
- File d'attente par agent/repo, audit local + VPS.
- Auth durcie : refresh court, révocation, HMAC payload, rotation manuelle.
- Tests de race/idempotence/expiration.
- Bouton `Approve once` uniquement pour intégrations qui supportent une décision programmatique documentée.

**V2 : hardening et confort.**

- Passkeys/WebAuthn, mTLS si le coût ops est justifié.
- Règles de policy locales : auto-approve lecture, demander mobile pour Bash/Edit/Write, deny automatique chemins sensibles.
- Support troisième agent via adaptateur.
- Observabilité : timeline de session, métriques latence, erreurs push.
- Mode secours : si VPS down, notifications locales Mac + Remote Tunnel.

Conclusion tranchée : **ne pas lancer le PRD sur "extension VS Code custom qui pilote Claude et Codex natifs". Lancer le PRD sur "mobile decision relay agent-agnostique", avec Remote Tunnels comme spike/fallback et Claude hooks comme première intégration réelle.**

#### Addendum Codex post-PRD — contournement MVP no-code

Après retour Arnaud ("peur de la complexité technique", hypothèse TeamViewer / contrôle Mac depuis iPhone), je durcis le verdict : **avant même le spike `code tunnel`, le premier MVP doit être no-code via remote desktop iPhone -> Mac**.

Option à tester en priorité :

1. **MVP 0 no-code** : remote desktop iPhone -> Mac.
2. **MVP 0 bis** : `code tunnel` / `vscode.dev` si remote desktop est trop lourd ou trop risqué.
3. **MVP custom `atelier-pilote`** uniquement si les deux contournements sont insuffisants au quotidien.

### Runbook MVP 0 — TeamViewer iPhone → Mac (pilotage VS Code en mobilité)

> Ce runbook documente le setup d'une solution **out-of-the-box no-code** permettant à Arnaud de prendre la main sur son Mac depuis son iPhone via TeamViewer pour répondre aux prompts d'autorisation et "next" demandés par les extensions VS Code Claude Code et Codex pendant ses déplacements. Validation opérationnelle finale : `## VALIDATION_UI` (Arnaud teste en réel sur son iPhone). NB : `META_FAST` skippe normalement `VALIDATION_UI` ; ici on l'accepte comme entorse en plus pour conserver un checkpoint terrain Arnaud — voir Verifications ci-dessous.

**Prérequis** :
- Mac d'Arnaud sous macOS récent, branché secteur, Wi-Fi/Ethernet stable.
- iPhone d'Arnaud avec connexion Wi-Fi ou 4G/5G.
- Compte TeamViewer (création gratuite : <https://www.teamviewer.com/>).

**Étape 1 — Installation TeamViewer Mac (côté hôte)** :
1. Télécharger TeamViewer pour macOS depuis <https://www.teamviewer.com/download/macos/>.
2. Installer le `.dmg`, glisser TeamViewer dans Applications, ouvrir.
3. Au premier lancement, créer un compte TeamViewer (ou se connecter à un compte existant).
4. Activer **2FA TeamViewer** sur le compte si la formule choisie le permet (Préférences compte → Sécurité).
5. Configurer **Unattended Access** :
   - Dans TeamViewer → menu `Connection` ou `Connexion` → `Setup Unattended Access` / `Configurer l'accès non surveillé`.
   - Définir un **mot de passe robuste ≥ 16 caractères** généré par un gestionnaire de mots de passe (1Password, Bitwarden, Trousseau iCloud, etc.). Stocker ce mot de passe dans le gestionnaire.
   - Lier ce Mac au compte TeamViewer pour qu'il apparaisse dans la liste des appareils.
6. Accorder les **permissions macOS requises** :
   - `Préférences Système` (ou `Réglages Système` selon macOS) → `Confidentialité et sécurité` → `Accessibilité` → activer `TeamViewer`.
   - `Confidentialité et sécurité` → `Enregistrement écran` → activer `TeamViewer`.
   - Redémarrer TeamViewer après changements de permissions si nécessaire.
7. **Empêcher la mise en veille** pendant les absences :
   - Soit `Réglages Système` → `Batterie` / `Économiseur d'énergie` → cocher `Empêcher la mise en veille automatique de l'ordinateur quand l'écran est éteint`.
   - Soit installer une app `caffeinate` GUI (ex. Amphetamine) ou utiliser CLI : `caffeinate -d -i &` à lancer avant départ.

**Étape 2 — Installation TeamViewer iPhone (côté client)** :
1. Installer **TeamViewer Remote Control** depuis l'App Store : <https://apps.apple.com/app/teamviewer-remote-control/id692035811>.
2. Ouvrir l'app, se connecter au **même compte TeamViewer** que celui du Mac.
3. Le Mac doit apparaître dans la liste `Mes appareils` / `My Computers`.

**Étape 3 — Test de prise de main** :
1. Depuis l'iPhone, taper sur le Mac dans la liste → `Connect` → entrer le mot de passe Unattended Access.
2. L'écran du Mac apparaît sur l'iPhone (en mode remote desktop).
3. Naviguer jusqu'à VS Code (déjà ouvert avec extensions Claude Code + Codex).
4. Tester un clic dans le panneau Claude Code et un clic dans le panneau Codex pour valider la précision du tap.

**Étape 4 — Sécurité minimale** (voir AC4 du PRD) :
- Mot de passe Unattended Access ≥ 16 caractères, jamais réutilisé.
- 2FA TeamViewer activé sur le compte.
- Désactivation préventive : si non utilisé pendant > 1 mois, quitter TeamViewer Mac (menu → `Quitter TeamViewer`) ou désactiver Unattended Access.
- Compte TeamViewer dédié à cet usage, pas partagé avec d'autres usages.

**Étape 5 — Désinstallation / réversibilité** :
- Mac : quitter TeamViewer → désinstaller l'app (glisser dans la corbeille + suppression données dans `~/Library`).
- iPhone : supprimer l'app TeamViewer Remote Control.
- Compte TeamViewer : supprimer l'appareil de la liste, ou supprimer le compte via le portail web TeamViewer.
