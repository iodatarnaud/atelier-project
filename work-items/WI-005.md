## CADRAGE

Goal:
Ajouter une couche explicite de défense en profondeur contre les XSS sur les interpolations HTML restantes dans les renders : aujourd'hui les `${id}`, `${color}`, `${type}`, `${status}`, `${priority}` sont interpolés dans des `onclick="..."`, `style="..."`, `class="..."`, `data-*="..."` sans échappement contextuel. Le risque concret est faible (validation amont via `normalizeImportedState`), mais la défense en profondeur manque : si un payload Gist échappe à la normalisation (régression future, type ajouté sans validation, etc.), une chaîne malicieuse peut s'évader d'un attribut HTML ou d'un string JS dans un onclick.

Persona impact:
- Arnaud — protection invisible mais cruciale : si un Gist se retrouve compromis (machine perso, fuite PAT) ou si un futur item du backlog ajoute un champ utilisateur non validé, l'app ne devient pas un vecteur XSS.
- Tout futur contributeur qui ajoute un nouveau champ rendu : le pattern d'escape contextuel évite l'oubli.

Context:
- Item backlog : ATE-9 « SEC-1 — Escape contextuel des interpolations HTML restantes » (Bug sécu, P2-P3, projet Atelier).
- **Brief technique fourni par Arnaud** : ajouter `escapeJs(str)` (escape `'` et `\` pour les chaînes JS dans `onclick`) et `escapeAttr(str)` (escape `"` pour les attributs HTML), puis les appliquer systématiquement.
- **Audit code (Claude, hors WI)** :
  - `escapeHtml(s)` existe déjà ([`index.html:5317`](index.html#L5317)), utilisé pour le contenu textuel (~30 sites).
  - Sites d'interpolation NON échappés (audit grep `\$\{` dans `onclick=`, `style=`, `class=`, `data-*=`, `title=`) :
    - `onclick="...'${id}'..."` (~12 sites — switchClient, cycleStatus, openItemDetail, etc.).
    - `style="background:${color}"` (~6 sites — pastilles client/sprint/epic).
    - `class="status-${item.status}"`, `class="prio-${item.priority}"`, `class="type-${item.type}"` (~10 sites).
    - `data-id="${id}"`, `data-section="${section.id}"`, `data-client-id="${c.id}"`, etc. (~8 sites).
    - `title="${TYPE_LABELS[item.type]}"` (champ statique mais bonne pratique).
  - Note : les lignes mentionnées dans le brief ATE-9 (2381, 2430, 2434, 2449, 2453, 2677, 2679-2687, 3316-3318) ont décalé depuis l'ajout WI-004 sync (v0.17.0). L'audit doit être ré-effectué exhaustivement à l'IMPLEMENTATION.
- Validation amont actuelle (`normalizeImportedState` lignes ~2640-2680) :
  - `id` : reconstruit via `_clampStr` ou `crypto.randomUUID()`, considéré sûr.
  - `color` : regex hex 6 chars (`#[0-9a-f]{6}`).
  - `type` / `status` / `priority` : whitelist enum.
  - Donc le risque actuel est résiduel mais réel si la normalisation a un trou (futur ajout de champ, branch oubliée).

Constraints:
- Pas de refonte du modèle de rendu (template strings → DOM API par exemple). Trop intrusif.
- Pas de touche à `normalizeImportedState` (la validation amont reste la 1ʳᵉ ligne de défense).
- Compatibilité Gist préservée (zéro impact sur le payload écrit).
- Pas de régression visuelle / fonctionnelle : les renders doivent produire exactement le même HTML que maintenant pour les payloads valides.

Out of scope:
- CSP (Content-Security-Policy) — ajouter `default-src 'self'; script-src 'self'` etc. : intrusif, casse les `onclick=` inline du projet, prévu dans un autre item backlog.
- Refonte des `onclick=` inline → event listeners délégués : refonte UI complète, hors-périmètre sécu.
- Sanitization CSS au-delà de l'escape de `"` dans les attributs (injection CSS via `style="background:red; behavior:url(xss.htc)"` reste couverte par la validation regex hex amont).
- Audit complet OWASP des autres vecteurs (DOM clobbering, prototype pollution, etc.) — ce WI cible uniquement les interpolations HTML/JS.

Open questions:

1. **Périmètre de couverture** — couvrir uniquement les sites listés dans le brief, ou tout site d'interpolation `${...}` dans contexte HTML/JS/attr ?
   - **(A) Exhaustif** : audit grep complet, patcher tous les sites trouvés (`onclick=`, `style=`, `class=`, `data-*=`, `title=`). Coût marginal vs ciblé : ~5 lignes max en plus.
   - **(B) Ciblé** : uniquement les lignes mentionnées dans le brief.

2. **`style="background:${color}"`** — la couleur est validée en amont (regex hex). On l'enrobe dans `escapeAttr` quand même, ou on assume la validation amont ?
   - **(I) Cohérence** : `escapeAttr(color)` partout. Coût zéro, défense en profondeur cohérente avec le reste.
   - **(II) Économie** : on assume `normalizeImportedState`. Pas d'overhead, mais inconsistant.

Decisions:
1. **Q1 → (A) Exhaustif.** Justif : les lignes du brief datent d'avant v0.17.0 (mes ajouts WI-004 ont décalé). Si on rate un site, on garde un trou de défense en profondeur. Coût marginal vs ciblé négligeable. Audit grep `\$\{` dans contextes attribut/JS = source de vérité unique.
2. **Q2 → (I) Cohérence.** Justif : on cherche exactement le pattern « défense en profondeur » → on ne fait pas confiance à la validation amont uniquement. `escapeAttr` empêche l'évasion d'attribut HTML. L'injection CSS pure (`; behavior:url(...)`) reste couverte par la regex hex de `normalizeImportedState`. Pas de refonte du modèle CSS dans ce WI.

Note canal : Arnaud m'a délégué l'arbitrage Q1+Q2 en chat (« je suis ok pour ton autonomie en fast track »), inscrit ici par Claude. Mode FAST_TRACK validé en chat également.

### Sécurité — défense en profondeur sur les interpolations HTML (SEC-1 / ATE-9)

Cette release durcit le rendu de l'app contre une catégorie d'XSS qui restait théoriquement exploitable si un payload Gist contournait la validation amont (`normalizeImportedState`). Aucun comportement utilisateur ne change.

#### Ce qui a été ajouté

Deux helpers d'échappement contextuel à côté de l'`escapeHtml` existant :

- **`escapeAttr(s)`** : alias sémantique d'`escapeHtml` à utiliser dans les valeurs d'attribut (`<div data-x="${escapeAttr(v)}">`). Les chars `& < > " '` sont neutralisés en entités HTML — impossible de fermer l'attribut hôte ni d'en injecter un nouveau.
- **`escapeJs(s)`** : encode en `\uHHHH` les chars dangereux (`'`, `"`, `\`, retours à la ligne, séparateurs Unicode, `<>&`) pour les interpolations dans une string JS encadrée elle-même par un attribut `onclick="..."` (ou `on*="..."`). Idempotent à travers le décodage HTML puis le parsing JS — la valeur reçue par le handler vaut l'original.

#### Ce qui a été couvert

Audit grep exhaustif des `${...}` dans les contextes attribut/JS de `index.html` :

- 32 sites `on*="...'${id}'..."` (switchClient, cycleStatus, openItemDetail, sprintActionMenu, etc.) → wrappés par `escapeJs`.
- 66 sites attributs (`data-*=`, `class=`, `style=`, `title=`, `value=`, `name=`, `for=`, `aria-*=`) → wrappés par `escapeAttr`.
- Idempotence assurée : les sites déjà entourés de `escapeHtml(...)` (contenu textuel) ne sont pas re-wrappés.

La validation amont via `normalizeImportedState` reste la 1ʳᵉ ligne de défense (regex hex sur les couleurs, whitelist d'enum sur les types/statuts/priorités, IDs reconstruits via `_clampStr`). Cette release ajoute la 2ᵉ ligne — si la 1ʳᵉ a un trou un jour, le rendu reste sûr.

#### Tests

5 nouveaux tests E2E dans `tests/security.spec.js` (`test.describe('Sécurité — escape contextuel attributs/JS (WI-005)')`) :

- Helpers `escapeJs` et `escapeAttr` vérifiés sur 23 cas (vide, null, alphanum sûr, vecteurs d'évasion `</script>` + `<img onerror=...>`).
- Injection directe d'un id de client malicieux (avec `'`, `"`, balises) en bypass de `normalizeImportedState` : aucun élément étranger ajouté au DOM, le handler reçoit l'id original.
- Injection directe d'une couleur malicieuse (tente d'évader l'attribut `style`) : inspection DOM via `getAttributeNames()` confirme qu'aucun handler `onload`/`onerror` n'est attaché, seuls les attributs légitimes (`class`, `style`) sont présents.
- Régression standard : un id alphanumérique normal continue d'ouvrir la modale détail (les helpers sont idempotents pour les valeurs sûres).

#### Compatibilité

- Aucun changement de schéma `lastSavedAt`, payload Gist identique.
- Aucun changement d'API utilisateur, comportement visuel inchangé.
- Aucun listener / handler global ajouté.
