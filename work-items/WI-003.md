## CADRAGE

Goal:
Décider quelle sémantique de couleur appliquer à la pastille ronde devant chaque item dans la vue Calendrier (`.cal-item-color`), suite au signalement Arnaud que la pastille apparaît grise alors que les items ont un Epic défini avec couleur.

Persona impact:
- L'utilisateur (Arnaud) qui consulte le calendrier mensuel multi-projets pour planifier sa charge.
- Confort visuel : reconnaître d'un coup d'œil la nature/appartenance d'un item sans devoir cliquer.

Context:
- Item backlog : ATE-21 « les WI dans le calendrier affichent un rond » (Bug, P2, échéance 30/04/2026, projet Atelier).
- Description Arnaud : « Pourquoi aujourd'hui des ronds gris sur les WI dans le calendrier pour le projet d'un client alors que les Epics sont définis sur les WI ? Quelle signification de ce rond gris ? Mettre la couleur de l'Epic est pour moi un bon choix UX. »
- **Audit code (Claude, hors WI)** : la pastille est rendue par `renderCalendarItem(it, withDateLabel)` à [`index.html:3931-3941`](index.html#L3931-L3941). La couleur est : `const color = it.clientColor || '#6c7888';`. Donc actuellement = **couleur du client/projet**, fallback gris (`#6c7888`) si non définie.
- Le projet "Atelier" semble ne pas avoir de couleur configurée → fallback gris visible.
- CSS : `.cal-item-color { width: 8px; height: 8px; border-radius: 50%; }` ([`index.html:1574-1579`](index.html#L1574-L1579)).
- Référence Epic ailleurs dans l'app :
  - Backlog (ligne 3389) : `epicDot` 10×10px de couleur Epic devant le titre de section.
  - Modale détail (ligne 3439, 3528) : `epic-chip` rectangulaire avec nom + couleur de l'Epic.
- Donc il existe déjà 2 visualisations Epic dans l'app, cohérence à maintenir.

Constraints:
- Pas de migration de schéma de données.
- Pas de modification de la logique D&D du calendrier (déjà couverte par WI-001).
- Garder la pastille discrete (8px), pas grossir le composant.
- Décision UX, pas un fix purement technique → arbitrage Arnaud nécessaire.

Out of scope:
- Refonte de la vue Calendrier.
- Ajout de paramétrage utilisateur (toggle « afficher couleur Epic vs Client »).
- Modification du `epic-chip` ou du `epicDot` du backlog (cohérence Epic ailleurs déjà OK).
- Imposer une couleur par défaut aux clients sans couleur (autre WI éventuel si on choisit option B).

Open questions:

1. **Sémantique cible de la pastille** — quelle info la pastille doit-elle véhiculer ?
   - **(A) Couleur Epic** : la pastille devient le marqueur du chantier transverse (Epic) auquel l'item appartient. Fallback gris si pas d'Epic. Cohérent avec `epicDot` du backlog. ➜ ce que demande Arnaud.
   - **(B) Couleur Client/projet** (statu quo, fix gris seulement) : la pastille reste l'indicateur de projet (utile en vue multi-projets). On corrige seulement le cas "client sans couleur" (forcer une couleur par défaut, ou alerter en CRUD client).
   - **(C) Hybride : Epic si défini, sinon Client** : meilleur des 2 mondes, mais introduit une ambiguïté visuelle (la même pastille verte peut signifier "Epic Vert" pour l'item X et "Projet Vert" pour l'item Y).
   - Arbitrer A / B / C.

2. **Cas item sans Epic** (si on retient A ou C) — comportement de la pastille :
   - **(a)** Gris neutre (`#6c7888`, comme aujourd'hui).
   - **(b)** Couleur du client/projet (option C).
   - **(c)** Pas de pastille du tout (espace réservé invisible).
   - **(d)** Pastille bordurée vide (cercle creux).

3. **Tooltip / accessibilité** — quand on survole la pastille, faut-il afficher le nom de l'Epic (ou du Client) en complément du `title` actuel qui montre `Client · Titre item` ?
   - **(a)** Garder le `title` actuel inchangé.
   - **(b)** Étendre le `title` à `Client · Epic · Titre item`.
   - **(c)** Ajouter un tooltip CSS pur sur la pastille seule.

4. **Migration / rétrocompatibilité** — si on retient l'option A, certains items historiques pourraient ne pas avoir d'Epic (donc pastille grise). Pas un blocker mais à noter.

Decisions:
1. **Sémantique : (A) Couleur Epic, définitivement.** La pastille devient le marqueur du chantier transverse (Epic). Cohérent avec `epicDot` du backlog et `epic-chip` de la modale détail. Le projet/client est déjà visible via le titre tronqué dans le tooltip et via le filtre projet du calendrier.
2. **Si pas d'Epic : (a) Gris neutre (`#6c7888`).** Le gris devient un signal honnête « pas d'Epic défini » plutôt qu'une couleur arbitraire trompeuse. Encourage à assigner un Epic.
3. **Tooltip : (b) Étendre le `title` à `Client · Epic · Titre item`.** Zero-cost, utile en vue multi-projets, garde le natif HTML (pas de tooltip CSS custom à maintenir).
4. **Migration : N/A.** Les items sans Epic existant verront simplement la pastille grise (= signal correct). Pas de tâche de fond, pas de migration de données.

Note canal : décisions transmises par Arnaud via chat (ergonomie pratique pour 4 questions courtes), inscrites par Claude dans la section. Per AC2 du protocole v2.1, la voie nominale est l'édition IDE par Arnaud + `next`. Si ce canal informel se répète, à observer en `RETROSPECTIVE` pour ré-arbitrage.

## PRD

Objective:
Remplacer la sémantique de la pastille colorée des items dans la vue Calendrier : passer de **couleur client/projet** (état actuel, avec fallback gris) à **couleur Epic** (avec fallback gris si pas d'Epic). Étendre le tooltip natif HTML pour inclure le nom de l'Epic. Décisions actées en `CADRAGE.Decisions` (A + a + b + N/A).

Current behavior:
- `getAllCalendarItems()` ([`index.html:3755-3763`](index.html#L3755-L3763)) enrichit chaque item avec `clientId`, `clientName`, `clientColor` mais **pas** avec `epicColor` ni `epicName`.
- `renderCalendarItem(it, withDateLabel)` ([`index.html:3931-3941`](index.html#L3931-L3941)) calcule `const color = it.clientColor || '#6c7888';` → la pastille est de la couleur du client, gris (`#6c7888`) si client sans couleur définie.
- Le tooltip (attribut `title`) est `${escapeHtml(it.clientName)} · ${escapeHtml(it.title)}` → `Client · Titre`.
- 3 zones d'affichage utilisent `renderCalendarItem` : cellules du mois (ligne 3884), section "En retard" (ligne 3859), section "Non planifiés" (ligne 3913).

Target behavior:
- `getAllCalendarItems()` enrichit en plus avec `epicColor` (couleur de l'Epic résolu via `it.epicId` dans `c.epics`) et `epicName`.
- `renderCalendarItem` calcule `const color = it.epicColor || '#6c7888';` → la pastille devient le marqueur du chantier transverse (Epic).
- Le tooltip devient `${escapeHtml(it.clientName)}${it.epicName ? ' · ' + escapeHtml(it.epicName) : ''} · ${escapeHtml(it.title)}` → `Client · Epic · Titre` si Epic, sinon `Client · Titre` (tooltip natif HTML, zero-cost).
- Comportement uniforme appliqué par construction aux 3 zones (une seule fonction modifiée).

User stories:
- **US1** — En tant qu'Arnaud consultant qui consulte la vue Calendrier multi-projets, je veux que la pastille de chaque item reflète son Epic, pour identifier visuellement à quel chantier transverse il appartient sans ouvrir la modale.
- **US2** — En tant qu'Arnaud, je veux que les items sans Epic affichent une pastille gris neutre, pour que le gris signale clairement « Epic non défini » et m'incite à en assigner un.
- **US3** — En tant qu'Arnaud, je veux que le tooltip au survol affiche `Client · Epic · Titre` (et `Client · Titre` si pas d'Epic), pour avoir le contexte complet sans clic.

Acceptance criteria:

### AC1 — Item avec Epic défini : pastille = couleur Epic
Given un item `it` dans la vue Calendrier dont `it.epicId` correspond à un Epic existant `e` dans le client courant
When la pastille (`.cal-item-color`) est rendue dans n'importe quelle zone (cellule mois, En retard, Non planifiés)
Then `style.background` de la pastille est égal à `e.color` (couleur hexadécimale de l'Epic)
And la pastille n'est plus jamais grise (`#6c7888`) tant que l'Epic existe.

### AC2 — Item sans Epic : pastille gris neutre
Given un item dont `it.epicId` est `null`, `undefined`, ou pointe vers un Epic supprimé/introuvable
When la pastille est rendue
Then `style.background` est `#6c7888` (gris neutre).
And aucune erreur JS n'est levée (résolution Epic défensive).

### AC3 — Tooltip étendu : Client · Epic · Titre
Given un item rendu dans une zone calendrier
When l'attribut `title` du `.cal-item` est généré
Then :
- Si l'item a un Epic résolu → `title = "<Client> · <Epic> · <Titre>"`
- Sinon → `title = "<Client> · <Titre>"` (comportement actuel, inchangé)
And le contenu est échappé via `escapeHtml` pour éviter toute injection.

### AC4 — Cohérence dans les 3 zones du calendrier
Given un item présent dans plusieurs sections du calendrier (par ex. un item avec `dueDate` passée et statut non terminé apparaît dans "En retard" + cellule du mois si jamais le mois affiché contient cette date)
When les pastilles sont rendues
Then la couleur ET le tooltip sont identiques dans toutes les sections (puisque `renderCalendarItem` est l'unique chemin de rendu).

### AC5 — Aucune régression de comportement
Given le comportement actuel des `cal-item` : drag & drop (WI-001), click pour ouvrir la modale (`openItemFromCalendar`), classe `done` (opacity 0.5 + barré), classe `overdue` (fond rouge transparent)
When le code est modifié pour la pastille et le tooltip
Then aucun de ces comportements n'est altéré : les attributs `draggable`, `data-client-id`, `data-item-id`, `onclick`, et les classes CSS restent identiques.

Out of scope:
- Imposer une couleur par défaut aux clients sans couleur (autre WI éventuel si besoin).
- Modifier la pastille ailleurs que dans le calendrier (`epicDot` du backlog, `epic-chip` de la modale détail restent inchangés).
- Ajouter un toggle utilisateur « pastille = Epic vs Client » (refusé en CADRAGE.Out of scope).
- Migration des items historiques sans Epic (le gris est volontaire et incite à assigner un Epic).
- Tests visuels (screenshot regression) : trop coûteux pour ce changement, on se contente de tests fonctionnels qui vérifient les attributs DOM.

Assumptions:
- Le champ `epicId` existe déjà sur tous les items (création obligatoire ou null).
- Les Epics ont un champ `color` non-null (vérifié via `epic-color-swatch` à la création).
- La fonction `renderCalendarItem` est le seul point de rendu des `cal-item` (validé par grep sur les 3 zones d'usage).
- Aucun test E2E existant n'asserte la couleur actuelle de la pastille = couleur client (à confirmer en TESTS).

Risks:
- **Régression visuelle** si un autre code ailleurs s'attend à la couleur client sur la pastille calendrier. Mitigation : grep exhaustif `cal-item-color` confirme un seul usage (CSS + render).
- **Tests E2E préexistants** qui asserteraient une couleur spécifique → casseront. Mitigation : audit en TESTS, ajustement si nécessaire (cause "test-side" per AC4 du protocole = loop interne TESTS, pas retour IMPL).
- **Confusion utilisateur** si un Epic et un Client ont par hasard la même couleur dans le panel. Acceptable : la palette `EPIC_COLORS` est volontairement limitée (6 couleurs), donc ce sera fréquent mais lisible (le titre + tooltip lèvent l'ambiguïté).
- **Performance** : la résolution `c.epics.find(e => e.id === it.epicId)` est appelée pour chaque item à chaque rendu. Cost négligeable (`epics` typiquement < 10 par client, items < 100). Mitigation : si besoin, mémoïser via une `Map` au début de `getAllCalendarItems` — overkill à ce stade.

### Vue Calendrier — pastille = couleur de l'Epic

La pastille ronde devant chaque item du calendrier reflète désormais la **couleur de l'Epic** auquel l'item appartient, plutôt que la couleur du projet. Cohérent avec le marquage Epic déjà présent dans le backlog (`epicDot`) et la modale détail (`epic-chip`).

#### Comportement

- Item avec un Epic défini → pastille de la couleur de l'Epic.
- Item sans Epic (ou avec un Epic supprimé) → pastille **gris neutre** (`#6c7888`). Signal honnête « Epic non défini » qui invite à en assigner un.
- Comportement uniforme dans les 3 zones du calendrier : cellules du mois, section « ⚠ En retard », section « Non planifiés ».

#### Tooltip enrichi

Le tooltip natif au survol d'un item affiche maintenant :

- `Client · Epic · Titre item` si l'item a un Epic ;
- `Client · Titre item` sinon (comportement actuel inchangé pour ce cas).

Utile en vue multi-projets pour avoir le contexte complet sans clic.

#### Tests

7 nouveaux tests Playwright dans `tests/calendrier.spec.js` (couleur Epic / fallback gris / Epic orphelin / tooltip avec & sans Epic / cohérence multi-zones / anti-régression D&D + click). Suite complète : 132 tests verts.

#### Améliorations à étudier (hors scope WI-003)

Lors de la validation, deux pistes ont été notées pour itérations futures :

- Couleur **hybride contextuelle** : projet en mode « Tous projets », Epic quand un projet est filtré (cohérent avec la sidebar qui n'affiche qu'un projet à la fois).
- **Synchroniser** la sélection de projet dans la sidebar avec le filtre projet du calendrier.

Ces idées peuvent faire l'objet d'items backlog dédiés.
