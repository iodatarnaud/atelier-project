## CADRAGE

Goal:
Concevoir une pipeline AI **sub-FAST_TRACK** dédiée aux **micro-ajustements du protocole AI lui-même** (modifs `ai-system/`, conventions de WI, helpers de doc protocole), sans impact code/app, pour faire passer la durée moyenne d'un tel WI de ~15-20 min (FAST_TRACK actuel) à **≤ 8 min** (cible Arnaud).

L'objectif n'est PAS de relâcher la rigueur — c'est d'enlever les phases qui n'apportent rien à un méta-WI sans code app (VALIDATION_UI sur du protocole texte, PATCH_NOTES pour une release qui n'existe pas, DOCS user-facing pour un changement non user-facing, etc.). Codex doit participer à la réflexion pour garantir un pipe **béton armé** qui ne crée pas de trou de gouvernance.

Persona impact:
- Arnaud — réduit la friction des allers-retours `next` ↔ Claude pour des tweaks de 5-10 lignes de doc protocole. Aujourd'hui chaque transition coûte ~30-60s (force-read + edit + output contract), multiplié par 7-8 phases = ~5 min de overhead pur.
- Claude / Codex — moins de cérémonial sur les WI triviaux, plus de focus sur les chantiers à vraie valeur (features, fixes, sécu).

## PRD

Objective:
Définir un 3ᵉ mode d'exécution `META_FAST` dans le protocole AI, applicable strictement aux WI qui ne touchent que `ai-system/` ou `work-items/` (pas de code app, pas de doc user-facing, pas de bump version). Cible mesurable : durée moyenne ≤ 8 min (vs 16 min mesurées sur WI-006).

Current behavior:
- 3 modes définis dans `00_AI_SYSTEM.md` ligne 239-269 : `STANDARD` (13 phases), `FAST_TRACK` (8 phases), pas de mode dédié aux méta-WI sans code app.
- Pour un méta-WI sans impact app (ex. WI-006), `FAST_TRACK` reste le mode applicable. Mais sa pipeline contient :
  - `TESTS` obligatoire → on lance `npm test` (3 min) alors que rien d'app n'est touché.
  - `VALIDATION_UI` owned by Arnaud → round-trip explicite alors qu'aucune UI à valider.
  - `PATCH_NOTES` rédigé alors qu'aucune `gh release` n'est créée (méta-WI variant degraded path).
  - `DOCS` no-op explicite mais reste une phase à transitionner.
- Résultat : ~8 round-trips `next` ↔ Claude + ~3 min de smoke test inutile + 2 commits métadata séparés = ~16 min pour un changement de 44 lignes de doc protocole.

Target behavior:
- Nouveau mode `META_FAST` déclaré dans le header (`Mode: META_FAST`) avec auto-vérification d'éligibilité au boot `CADRAGE`.
- Pipeline réduite à **5 phases** : `CADRAGE → IMPLEMENTATION → MERGE_RELEASE → CLEANUP → DONE`.
- Phases skippées : `PRD`, `PLAN_REVIEW`, `TESTS`, `CODE_REVIEW`, `VALIDATION_UI`, `PATCH_NOTES`, `DOCS`, `RETROSPECTIVE` (8 phases skippées).
- Vérifs textuelles (équivalent TESTS pour ai-system/) obligatoires en sous-section `IMPLEMENTATION.Verifications:` (liste de greps PASS sur les fichiers touchés).
- Output contract identique (6 champs) — cohérence totale.
- Garde-fou scope creep : si pendant `IMPLEMENTATION` un fichier hors `ai-system/` ou `work-items/` doit être touché, ou si le diff dépasse 30 lignes, STOP, BLOCKER, escalade `Mode: META_FAST → FAST_TRACK` (ou `STANDARD` si scope explose).

User stories:
- **US1** — En tant qu'Arnaud, quand je fais un micro-tweak du protocole (ex. ajouter un champ dans le template, renommer une convention), je veux que le WI dure ≤ 8 min calendaires, pas 16+.
- **US2** — En tant qu'Arnaud, je veux conserver la garantie que rien d'app n'est touché par accident dans un méta-WI (le check out-of-scope du `MERGE_RELEASE` reste actif).
- **US3** — En tant que Claude/Codex, je veux que le protocole me dise explicitement quand `META_FAST` est éligible vs quand il faut escalader, pas avoir à deviner.

Acceptance criteria:

### AC1 — Mode `META_FAST` ajouté à l'enum
Given le protocole v2.3
When un agent lit `00_AI_SYSTEM.md → ## Execution modes`
Then `Mode: STANDARD | FAST_TRACK | META_FAST` est documenté.
And la chaîne `META_FAST: CADRAGE → IMPLEMENTATION → MERGE_RELEASE → CLEANUP → DONE` est explicitement listée.
And les phases skippées (`PRD`, `PLAN_REVIEW`, `TESTS`, `CODE_REVIEW`, `VALIDATION_UI`, `PATCH_NOTES`, `DOCS`, `RETROSPECTIVE`) sont nommément listées.

### AC2 — Critères d'éligibilité `META_FAST`
Given un agent au boot `CADRAGE` d'un WI marqué `Mode: META_FAST`
When il vérifie l'éligibilité
Then les critères suivants sont tous remplis :
- `Risk: LOW`
- Modifs strictement dans `ai-system/` ou `work-items/` (aucun fichier `index.html`, `tests/`, `MANUEL.md`, `README.md`, `SMOKE-TEST.md`, `CLAUDE.md`, `package.json`, `playwright.config.js`).
- Pas de bump version `index.html`.
- Scope diff prévisible < 30 lignes.
And si l'un des critères n'est pas rempli au boot CADRAGE → BLOCKER, escalade `Mode: FAST_TRACK` ou `STANDARD` selon ampleur.

### AC3 — Vérifs textuelles intégrées à IMPLEMENTATION
Given un méta-WI en `META_FAST` exécutant `IMPLEMENTATION`
When Claude finalise les modifs `ai-system/`
Then la sous-section `## IMPLEMENTATION.Verifications:` est remplie avec une liste de checks `grep` sur les fichiers touchés (présence des sections clés, non-régression du vocabulaire, comptage des occurrences attendues), avec verdict PASS/FAIL pour chaque.
And aucun `npm test` n'est lancé (la pipeline n'a pas de phase TESTS).
And si un check FAIL → STOP, BLOCKER, fix avant transition vers MERGE_RELEASE.

### AC4 — Output contract identique
Given un agent en `META_FAST` finalisant une phase
When il émet son output
Then le contrat 6-field est inchangé : `PHASE_EXECUTED / STATUS_UPDATED_TO / QUALITY_CHECK / BLOCKERS / NEXT_PHASE / NEXT_PHASE_OWNER`.
And aucun champ n'est ajouté ni supprimé pour `META_FAST`.

### AC5 — Trigger explicite par déclaration `Mode:`
Given un nouveau WI au boot `CADRAGE`
When Arnaud (ou Claude au nom d'Arnaud) déclare `Mode: META_FAST` dans le header
Then Claude vérifie automatiquement l'éligibilité (AC2) au boot.
And aucune auto-promotion `FAST_TRACK → META_FAST` n'a lieu (le mode est strictement déclaratif).

### AC6 — Garde-fou scope creep en cours d'exécution
Given un méta-WI en `META_FAST` au stade `IMPLEMENTATION`
When une modification hors `ai-system/` / `work-items/` devient nécessaire OU le diff cumulé dépasse 30 lignes
Then STOP, BLOCKER `meta-fast scope exceeded: <raison>`.
And `Mode` est ré-évalué : si scope reste raisonnable → escalade `FAST_TRACK` (ré-injection de TESTS / VALIDATION_UI / PATCH_NOTES / DOCS) ; sinon `STANDARD` (PRD + PLAN_REVIEW + CODE_REVIEW + RETROSPECTIVE ré-injectés).
And la sous-section `## IMPLEMENTATION` repart en mode `Révision après escalade <META_FAST → FAST_TRACK|STANDARD>` (Re-opened phase rule).

### AC7 — `MERGE_RELEASE` méta-WI degraded path conservé
Given un méta-WI en `META_FAST` au stade `MERGE_RELEASE`
When Claude exécute la phase
Then le degraded path existant (`MERGE_RELEASE — meta-WI variant`) s'applique : commit feature → checkout main → merge `--no-ff` → push, **sans `gh release create`** ni bump `index.html`.
And la section `## MERGE_RELEASE` documente `Version bumped: N/A (méta-WI)` et `Release URL: N/A`.

### AC8 — Section `## TIMING` respectée même en `META_FAST`
Given un méta-WI en `META_FAST`
When le WI est créé puis finalisé
Then `Opened` est posé au boot `CADRAGE`, `Closed` au passage `Status: DONE`, `Duration` calculée — selon la convention v2.2 introduite par WI-006.
And la cible `Duration ≤ 8min` est documentée comme objectif (pas un BLOCKER, juste un retex).

### AC9 — Cas Arnaud-channel adapté à `META_FAST`
Given que `META_FAST` skippe `VALIDATION_UI`
When Arnaud dit `next` après une phase appartenant à Claude (CADRAGE → IMPLEMENTATION par exemple)
Then aucun BLOCKER `<section> empty (Arnaud channel)` ne s'applique car Arnaud n'a pas de phase owned dans `META_FAST`.
And la `CADRAGE.Decisions` reste owned par Arnaud (il pose les décisions ou délègue explicitement) — c'est le seul canal Arnaud du mode `META_FAST`.

Out of scope:
- Auto-détection (sans déclaration `Mode:`) qui promote un WI en `META_FAST`. On veut un opt-in explicite.
- Mesure / lint automatique de la durée (alerte si Duration > 8min). Retex manuel suffit pour l'instant.
- Fusion ou raccourcissement du contrat 6-field. Hors-scope.
- Modes hybrides (`META_FAST_WITH_CODEX_GATE`). Si Arnaud doute du scope, il déclare `FAST_TRACK` directement.
- Tests automatisés du protocole lui-même (qu'un WI marqué `META_FAST` mais touchant `index.html` soit auto-rejeté). Le check humain au boot CADRAGE suffit.

Assumptions:
- Les vérifs textuelles `grep` sur `ai-system/` (10-15 secondes en CLI) sont une couverture équivalente à `npm test` (3 min) pour un changement strict ai-system/ — confirmé par WI-002 et WI-006.
- Le seuil 30 lignes diff est cohérent avec FAST_TRACK (50 lignes) et reflète qu'un méta-WI typique change 1-3 lignes par fichier sur 4 fichiers ai-system/ max.
- Codex n'est PAS bloqué par l'absence de `PLAN_REVIEW` / `CODE_REVIEW` en `META_FAST` : il peut intervenir ad-hoc si Arnaud le déclenche, ou le mode est escaladé en `FAST_TRACK` au moindre doute.

Risks:
- **R-A** Trou de gouvernance : un vrai bug glisse via un méta-WI qui paraissait safe. Mitigation : garde-fou AC6 + check `MERGE_RELEASE — out-of-scope` toujours actif + Codex peut être déclenché par escalade.
- **R-B** Sur-utilisation de `META_FAST` : Claude/Arnaud y mettent des changements borderline. Mitigation : critères AC2 stricts + escalade obligatoire AC6.
- **R-C** Duration cible (≤ 8 min) jamais atteinte en pratique. Mitigation : retex documenté (TIMING obligatoire AC8). Si systématiquement > 8 min après 3-5 méta-WI, ré-évaluer le scope skip de phases.
- **R-D** Cas edge : un WI qui touche `ai-system/` ET `work-items/templates/` (n'existe pas encore mais pourrait). Mitigation : AC2 inclut explicitement `work-items/`. Précisé en `META_FAST`.
- **R-E** Codex se sent court-circuité s'il n'est plus dans les méta-WI mineurs. Mitigation : c'est l'effet voulu (gain de vélocité), Codex reste obligatoire sur tout WI app et tous les `STANDARD` méta-WI (cf. WI-002 / WI-007).
